# I1: `verifyFileWithFallback` の命名と実装の乖離

- **重大度**: 🔵 Info
- **領域**: 命名 (`internal/verification`)
- **影響コマンド**: `record`, `verify`, `runner`

## 問題

`internal/verification/manager.go` 付近に存在する `verifyFileWithFallback` (類似名称の関数) の **名称** は "fallback があるかのような印象" を与えるが、実装はそうではない (あるいは期待される fallback 挙動と食い違う) 可能性がある。

セキュリティ上のバグではなく、**命名のミスリード** によって将来の保守者が誤った前提でコードを変更するリスクがあるため Info レベルとして記録する。

## 影響

- セキュリティ上の直接的な影響はない。
- ただし検証ロジックは本プロジェクトの中核防御層であり、関数名による誤解は **誤った修正** を誘発する。特に「fallback があるならこのエラーは握り潰して良いだろう」という判断を呼びかねない。

## 修正方針

### 案 A: リネーム

実装内容を正確に表す名前に変更する。例:

- 実際に fallback があるなら `verifyFileWithLegacyFallback` のように **何にフォールバックするか** を明示。
- fallback が無いなら `verifyFileStrict` や単に `verifyFile` にリネーム。

### 案 B: コメント追加

関数先頭に「ここで言う fallback とは何を指すか、何にはフォールバックしないか」を明記。

### 推奨

**案 A**。コメントは読まれない可能性があるが、名前は呼び出し側のコード全体で目に入る。本プロジェクトはセキュリティクリティカルであり、検証系の命名は厳格に実装と一致させるべき。

## 追加調査事項

- 本項目は静的解析中に気付いた違和感のメモであり、実装詳細の厳密な追跡は未実施。
- 実際にリネームを行う前に、関数の呼び出し側すべてと実装を再確認し、何がどう fallback しているのか (あるいはしていないのか) を明確化する必要がある。
