> 本書は 0142（判断軸2: 宛先パス信頼区分）で導入したオペランド抽出層の集中リファクタを扱う後続タスクの要件である。

## 1. 背景と目的

本書で使う用語:

- オペランド: コマンドが実際に読み書きする対象（パス）。0142 の `Operands` に対応する。
- 引数付きフラグ: 引数（多くは path の値）を取るフラグ。例 `cp -t DIR`、`dd of=FILE`。
- 真偽フラグ: 引数を取らない on/off のフラグ。例 `rm -f`。
- 再帰フラグ: 引数を取らないフラグのうち、再帰を指示するもの。例 `cp -R`。0142 のリスク判定で特別に扱う。
- 引数省略可（optional-arg）: 引数を付けても付けなくてもよいフラグ。例 `tar --one-top-level`。
- 非フラグ引数: フラグでもフラグの値でもない引数（コマンドが操作する対象ファイルなど）。

0142 は、ファイル操作コマンド（`cp`/`mv`/`rm`/`ln`/`tar`/`dd`/`chmod`/`install` 等、約 26 コマンド）が読み書きする対象
（オペランド）を抽出する `ClassifyDestinationZone` を実装した。抽出は現状、コマンドごとに個別実装された抽出処理として作られて
おり、各抽出処理が独自にフラグの知識（どのフラグが値を取るか、真偽か、再帰か、mode/ACL の値内文法、不完全形の fail-closed
規則）を持つ。

この設計は 0142 のレビューで、コマンドラインフラグ解析の取りこぼしを 4 ラウンドにわたり繰り返し顕在化させた。代表例:

- フラグの別名（表記違い）の取りこぼし: `-t` は処理するが `--target-directory` を見落とす（`ln`/`install` 等）。
- 引数省略可フラグの誤解析: `--one-top-level`（GNU tar、引数は省略可）を引数必須フラグ扱いし、後続の語を誤って値として消費する。
- 値内文法の不完全さ: `chmod` シンボリックモードの setuid 検知（`u+xs`）・`setfacl` の default ACL（`default:g:...:w`）。
- 不完全形での fail-closed の不統一: `-t` 付きでソース無し・必須引数欠落などの形で、抽出処理ごとに倒し方がばらつく。

重要な前提（0142 §3.2 で確定済み）: セキュリティ保証は抽出精度ではなく fail-closed な `Recognized` contract が担う。
共通スキャナは語を暗黙に捨てず、未知/曖昧なら `Recognized=false`→High＋既存判定を残す。よって解析の不備は過剰分類
（fail-closed）に縮退し、実害のある fail-open は (1) スキャナが値の形を取り落とす、(2) 抽出処理が捕捉済みの path 値を無視する
／値内パーサが不完全、の狭い 2 経路のみで、0142 で個別修正済みである。

したがって本タスクの目的は、新たなセキュリティ穴を塞ぐことではなく、再発の温床である「コマンドごとに独自実装した
引数パーサ」を構造的に解消することにある。具体的には、(1) 宣言的なコマンド別フラグ仕様、(2) それを消費する単一の汎用
getopt パーサ、(3) 「path を運ぶ引数付きフラグは必ず オペランドとして扱われる」ことを保証する完全性メタテスト、を導入し、
別名／引数省略可／fail-closed の統一の各クラスを、データ追加 1 箇所で完結する形に落とし込む。

### 1.1 ゴール

- 抽出処理の引数解析ロジックを単一の getopt パーサに集約し、コマンド固有部分を、宣言的データと役割を割り当てる薄い関数に縮小する。
- フラグの表記追加・引数省略可の扱いを、コード分岐ではなくデータ変更で完結させる。
- 「引数付きフラグの取りこぼし」「未分類の path 値」をメタテストで機械的に検出し、レビュー依存の網羅性確認をやめる。
- 0142 の観測可能な挙動（リスクレベル・`Operands`・`ReasonCodes`・`Recognized`・`Applies`）を完全に保存する。

## 2. スコープ

- **In**:
  - **F-001**: コマンド別の宣言的フラグ仕様（値／真偽／再帰の種別・すべての表記・引数省略可かどうかの区別・オペランドの役割割り当て規則）
  - **F-002**: 単一の汎用 getopt パーサ（`=value`・短縮フラグに続けて書いた値・短縮フラグの連結・`--`・引数省略可・別名の正規化）
  - **F-003**: 完全性メタテスト（path を運ぶ引数付きフラグが オペランドとして扱われることの保証・未分類の引数付きフラグの検出）
  - **F-004**: 既存挙動の保存（0142 全テスト緑・LocationResult 不変）
  - **F-005**: fail-closed `Recognized` contract の保存
- **Out**:
  - 判断軸2 のレベル規則・区分判定・操作固有の下限の変更: trust-critical/ordinary/safe-zone/unresolved の対応、
    操作固有の下限（権限付与・dd デバイス・再帰・機密複製等）、コマンド別特則は 0142 で確定済み。本タスクは挙動保存であり
    これらの意味を変えない（[0142/01_requirements.md](../0142_axis2_destination_zoning/01_requirements.md) F-001〜F-005）。
  - 専用リゾルバ（`ResolveOperandPath`）・Trusted 述語の変更: 0142 PR-2 で確定。本タスクの対象外。
  - 新規ファイル操作コマンドの追加: 本タスクは既存 26 コマンドのリファクタに閉じる。新コマンド追加は宣言的仕様への
    エントリ追加で容易になるが、追加自体は本タスクのスコープ外。
  - データ送信書込形（`KindDataTransferWrite`）の意味変更: 0142 P5 で確定する抽出規則を保存する（表記／パーサの一元化対象には含む）。
  - `evaluateDimensions` への組み込み・config・identity 注入: 0142 P4/P6 で確定。本タスクは抽出層の内部リファクタに閉じる。

## 3. 機能要件と受け入れ基準

### F-001: 宣言的コマンドフラグ仕様

各対象コマンドのフラグ文法を、コード分岐ではなく単一の宣言的データ構造として表現する。フラグの表記・種別・オペランドの
役割割り当て規則を 1 箇所に集約し、表記追加やコマンド追加をデータ変更で完結させる。

**Acceptance Criteria**:
- **AC-01**: 各対象コマンドのフラグ仕様が単一の宣言的テーブルで定義される。引数付きフラグは短縮形・長形式のすべての表記を 1 エントリに
  まとめ、引数が必須か省略可（optional-arg）かを区別して持つ。真偽フラグ・再帰フラグ・引数付きフラグが型／データ上で区別される。
- **AC-02**: フラグの表記追加（例: 既存の引数付きフラグへの別名追加）が仕様データ 1 箇所の変更で完結し、抽出ロジックのコード
  分岐追加を要さない（テストで、別名を追加すると当該別名経由の値が取得できるようになることを表明）。

### F-002: 単一の汎用 getopt パーサ

宣言的仕様を消費する単一のパーサが、全コマンドの argv を統一的に解析する。GNU/util-linux のフラグ形式を一元的に扱う。

**Acceptance Criteria**:
- **AC-03**: 単一パーサが次の形式を一元処理する: `--flag=value`、短縮フラグに続けて書いた値（`-C/usr`）、短縮フラグの連結
  （`-rf`。末尾が引数付きフラグなら、続けて書いた値・後続の語の値も含む）、`--`（オプション終端）、引数省略可（`--flag` 単独と
  `--flag=value`）。各形式が表駆動テストで検証される。
- **AC-04**: パーサは語を暗黙に捨てない。argv の各語は「既知フラグ」「捕捉した値」「非フラグ引数」のいずれかに
  分類される。未知フラグ、または引数付きフラグの値欠落は `Recognized=false`（fail-closed）。
- **AC-05**: 別名の正規化により、抽出処理は引数付きフラグの値を正規キー 1 つで取得できる。同一フラグの表記違い（`-t` と
  `--target-directory` 等）で値の取りこぼしが構造的に発生しない（表記違いの両形が同一の抽出結果を返すことをテストで表明）。
- **AC-06**: 引数省略可フラグは、引数を付随形（`--flag=DIR`）でのみ取り、単独形（`--flag`）では後続の語を消費しない
  （`tar --one-top-level -xf a.tar` が誤解析されず `Recognized=true`）。短縮形の引数省略可も付随形（`-oVALUE`）でのみ
  引数を取り、分離した後続の語は決して引数として消費しない（GNU getopt 準拠）。

### F-003: 完全性メタテスト

「引数付きフラグを宣言したが抽出処理がその値（path）を使い忘れる」fail-open クラスを、レビューでなくメタテストで機械的に検出する。

**Acceptance Criteria**:
- **AC-07**: 各コマンド仕様で宣言された引数付きフラグは、(a) オペランド（解決対象パス）として消費される、または (b) 非 path
  （値を無視してよい）と明示的に分類される、のいずれかであることをメタテストが検証する。いずれにも分類されない引数付きフラグ
  が存在するとメタテストが失敗する。
- **AC-08**: 0142 のレビューや過去のルールで顕在化した以下の代表ケースが回帰テストとして緑である。
  - フラグの別名: `-t` と `--target-directory` 等の表記違いで値を取りこぼさない。
  - 引数省略可: `--one-top-level`（`tar --one-top-level -xf a.tar`）を引数必須フラグ扱いして後続の語を誤消費しない。
  - `sed -e`: スクリプトフラグの値（スクリプト）を後続のファイル引数と取り違えない。
  - `chmod` シンボリック権限: カンマ区切りの各句を operator（`+`/`-`/`=`）と権限文字で評価し、`u+xs`・`u=rws` 等の結合
    setuid を検出する一方、`u-s` 等の除去を昇格と誤検出しない（`strings.Contains` 等の部分一致で判定しない）。
  - `setfacl` default ACL: `default:`/`d:` 接頭辞の有無でフィールド添字をシフトして対象（user/group/other）と権限を特定し、
    書込付与を取りこぼさない。
  - `chown`/`chgrp`: `--from`（フィルタ・非フラグ引数を残す）と `--reference`（非フラグ引数なし）の区別。
  - `ln`: シンボリック（`-s`/`--symbolic`、リンク親基準）とハードリンク（CWD 基準）の相対パス解決の違い。
  - `tar`: 第1の語（および `-` で始まる語）に限定したモード解析。

### F-004: 既存挙動の保存（挙動を変えないリファクタ）

本タスクはリファクタであり、観測可能な挙動を一切変えない。

**Acceptance Criteria**:
- **AC-09**: リファクタ前後で 0142 の全テスト（`internal/runner/base/security/destination_zoning_test.go`・
  `internal/runner/base/security/operand_path_resolver_test.go`・関連統合テスト）が変更なしで緑である。テストの期待値変更を
  伴う場合は挙動変更とみなし、本タスクの不適合とする。
- **AC-10**: 26 対象コマンド×代表フラグの表駆動テストで、各入力に対する `LocationResult`（`Applies`/`Recognized`/`Level`/
  `Operands`（Index/Raw/Resolved/Zone/Role/MatchedCritical/Trusted）/`ReasonCodes`）がリファクタ前と同一である
  （挙動差分ゼロ）。

### F-005: fail-closed 保証の保存

0142 の安全保証である fail-closed `Recognized` contract を保存する。

**Acceptance Criteria**:
- **AC-11**: 未知/曖昧形・解決不能オペランド・引数付きフラグの値欠落・不完全形（必須の非フラグ引数欠落）は、リファクタ後も
  `Recognized=false` となり High 下限へ倒れる（fail-closed）。語を暗黙に捨てる経路が存在しないことを、AC-04 の表明と
  併せて担保する。

## 4. 非機能要件

- **NF-001**: `make fmt`・`make test`・`make lint` が成功する。`./internal/runner/...` がコンパイル・緑であること。
- **NF-002**: 保守性: 新規ファイル操作コマンドの追加が、宣言的仕様エントリ 1 つ＋（必要なら）薄い役割割り当ての追加で完結する
  （コマンドごとの独自パーサ実装を要さない）ことを、追加例で示す。抽出層の引数解析の重複が削減される。
- **NF-003**: 決定性・read-only を維持する（live identity 不参照・副作用なし。0142 NF-003 を継承）。リファクタは解析の構造のみを
  変え、解決・区分判定・identity 注入の純粋性を変えない。

## 5. スコープ外の根拠

- 挙動変更を伴う改善は本タスクで行わない: 例えば「より多くのフラグを精密解析して過剰分類（fail-closed）を減らす」改善は
  価値があるが、挙動保存（F-004）と両立しないため別タスクとする。本タスクの完了基準は「同一挙動・構造のみ刷新」である。
- 判断軸2 の意味論・0143 の監査出力は対象外: レベル規則・区分・下限・コマンド別特則（0142）と logger 出力・文書（0143）は
  既存の所掌に従い、本タスクは抽出層の内部実装にのみ介入する。

## 6. 成功基準（要約）

- AC-01〜AC-11 が test/static で緑。
- 0142 の全既存テストが無改変で緑（挙動保存の最重要証拠）。
- 単一 getopt パーサ＋宣言的仕様＋完全性メタテストにより、別名／引数省略可／fail-closed の統一／未分類 path 値の
  各クラスが構造的に再発不能になっている。
