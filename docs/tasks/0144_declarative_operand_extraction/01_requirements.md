> 本書は 0142（判断軸2: 宛先パス信頼区分）で導入したオペランド抽出層の**集中リファクタ**を扱う後続タスクの要件である。

## 1. 背景と目的

0142 は、ファイル操作コマンド（`cp`/`mv`/`rm`/`ln`/`tar`/`dd`/`chmod`/`install` 等、約 26 コマンド）の作用オペランドを
抽出する `ClassifyDestinationZone` を実装した。抽出は現状、**コマンドごとの専用（bespoke）抽出器**として実装されており、
各抽出器が独自にフラグ知識（どのフラグが値を取るか、bool か、再帰か、mode/ACL の値内文法、不完全形の fail-closed 規則）を
持つ。

この設計は 0142 のレビューで、コマンドラインフラグ解析の取りこぼしを**4 ラウンドにわたり繰り返し**顕在化させた。代表例:

- **エイリアス綴りの取りこぼし**: `-t` は処理するが `--target-directory` を見落とす（`ln`/`install` 等）。
- **オプション引数フラグの誤解析**: `--one-top-level`（GNU tar、引数は任意）を必須値フラグ扱いし後続トークンを誤消費。
- **値内文法の不完全さ**: `chmod` シンボリックモードの setuid 検知（`u+xs`）・`setfacl` の default ACL（`default:g:...:w`）。
- **不完全形の fail-closed 一貫性**: `-t` 付きでソース無し・必須引数欠落の各形で抽出器ごとに倒し方がばらつく。

**重要な前提（0142 §3.2 で確定済み）**: セキュリティ保証は抽出精度ではなく **fail-closed な `Recognized` contract** が担う。
共通スキャナはトークンを暗黙に捨てず、未知/曖昧なら `Recognized=false`→High＋既存判定を残す。よって解析の不備は**過剰分類
（fail-closed）に縮退**し、実害のある fail-open は (1) スキャナが値形を取り落とす、(2) 抽出器が捕捉済みの path 値を無視する
／値内パーサが不完全、の狭い 2 経路のみで、0142 で個別修正済みである。

したがって本タスクの目的は、**新たなセキュリティ穴を塞ぐことではなく、再発の温床である「コマンドごとに独自実装した
引数パーサ」を構造的に解消すること**にある。具体的には、(1) 宣言的なコマンド別フラグ仕様、(2) それを消費する単一の汎用
getopt パーサ、(3) 「path を運ぶ value flag は必ず operand 化される」ことを保証する完全性メタテスト、を導入し、
エイリアス／オプション引数／fail-closed 一貫性の各クラスを**データ追加 1 箇所で完結する形**に落とし込む。

### 1.1 ゴール

- 抽出器の引数解析ロジックを単一の getopt パーサに集約し、コマンド固有部分を**宣言的データ＋ロール割当の薄い関数**に縮小する。
- フラグの綴り追加・オプション引数の扱いを**コード分岐ではなくデータ変更**で完結させる。
- 「value flag の取りこぼし」「未分類の path 値」を**メタテストで機械的に検出**し、レビュー依存の網羅性確認をやめる。
- 0142 の観測可能な挙動（リスクレベル・`Operands`・`ReasonCodes`・`Recognized`・`Applies`）を**完全に保存**する。

## 2. スコープ

- **In**:
  - **F-001**: コマンド別の宣言的フラグ仕様（value/bool/recursion・全綴り・optional-arg 標識・operand ロール規則）
  - **F-002**: 単一の汎用 getopt パーサ（`=value`・付随短縮値・短縮クラスタ・`--`・optional-arg・エイリアス正規化）
  - **F-003**: 完全性メタテスト（path を運ぶ value flag の operand 化保証・未分類 value flag の検出）
  - **F-004**: 既存挙動の保存（0142 全テスト緑・LocationResult 不変）
  - **F-005**: fail-closed `Recognized` contract の保存
- **Out**:
  - **判断軸2 のレベル規則・区分判定・操作固有の下限の変更**: trust-critical/ordinary/safe-zone/unresolved の対応、
    操作固有の下限（権限付与・dd デバイス・再帰・機密複製等）、コマンド別特則は 0142 で確定済み。本タスクは**挙動保存**であり
    これらの意味を変えない（[0142/01_requirements.md](../0142_axis2_destination_zoning/01_requirements.md) F-001〜F-005）。
  - **専用リゾルバ（`ResolveOperandPath`）・Trusted 述語の変更**: 0142 PR-2 で確定。本タスクの対象外。
  - **新規ファイル操作コマンドの追加**: 本タスクは既存 26 コマンドのリファクタに閉じる。新コマンド追加は宣言的仕様への
    エントリ追加で容易になるが、追加自体は本タスクのスコープ外。
  - **データ送信書込形（`KindDataTransferWrite`）の意味変更**: 0142 P5 で確定する抽出規則を保存する（綴り/パーサの一元化対象には含む）。
  - **`evaluateDimensions` への組み込み・config・identity 注入**: 0142 P4/P6 で確定。本タスクは抽出層の内部リファクタに閉じる。

## 3. 機能要件と受け入れ基準

### F-001: 宣言的コマンドフラグ仕様

各対象コマンドのフラグ文法を、コード分岐ではなく**単一の宣言的データ構造**として表現する。フラグの綴り・種別・operand
ロール規則を 1 箇所に集約し、綴り追加やコマンド追加をデータ変更で完結させる。

**Acceptance Criteria**:
- **AC-01**: 各対象コマンドのフラグ仕様が単一の宣言的テーブルで定義される。value flag は短縮形・長形式の**全綴りを 1 エントリ**に
  まとめ、引数が必須か任意（optional-arg）かを標識として持つ。bool flag・recursion flag・value flag が型/データ上で区別される。
- **AC-02**: フラグの綴り追加（例: 既存 value flag への別名追加）が**仕様データ 1 箇所の変更で完結**し、抽出ロジックのコード
  分岐追加を要さない（テストで、別名を追加すると当該別名経由の値が取得できるようになることを表明）。

### F-002: 単一の汎用 getopt パーサ

宣言的仕様を消費する**単一のパーサ**が、全コマンドの argv を統一的に解析する。GNU/util-linux のフラグ形式を一元的に扱う。

**Acceptance Criteria**:
- **AC-03**: 単一パーサが次の形式を一元処理する: `--flag=value`、付随短縮値（`-C/usr`）、短縮クラスタ（`-rf`、末尾 value flag の
  付随値・後続トークン値を含む）、`--`（オプション終端）、optional-arg（`--flag` 単独と `--flag=value`）。各形式が表駆動テストで
  検証される。
- **AC-04**: パーサは**トークンを暗黙に捨てない**。各 argv トークンは「既知フラグ」「捕捉した値」「positional」のいずれかに
  分類される。未知フラグ、または値フラグの値欠落は `Recognized=false`（fail-closed）。
- **AC-05**: エイリアス正規化により、抽出器は value flag の値を**正規キー 1 つ**で取得できる。同一フラグの綴り違い（`-t` と
  `--target-directory` 等）で値の取りこぼしが構造的に発生しない（綴り違いの両形が同一の抽出結果を返すことをテストで表明）。
- **AC-06**: optional-arg フラグは、引数を付随形（`--flag=DIR`）でのみ取り、単独形（`--flag`）では後続トークンを消費しない
  （`tar --one-top-level -xf a.tar` が誤解析されず `Recognized=true`）。

### F-003: 完全性メタテスト

「value flag を宣言したが抽出器がその値（path）を使い忘れる」fail-open クラスを、レビューでなく**メタテストで機械的に検出**する。

**Acceptance Criteria**:
- **AC-07**: 各コマンド仕様で宣言された value flag は、(a) operand（解決対象パス）として消費される、または (b) 非 path
  （値を無視してよい）と**明示的に分類**される、のいずれかであることをメタテストが検証する。いずれにも分類されない value flag
  が存在するとメタテストが失敗する。
- **AC-08**: 0142 のレビューや過去のルールで顕在化した代表ケース（エイリアス綴り `--target-directory`、optional-arg
  `--one-top-level`、`sed -e` のスクリプトフラグ、setuid 結合 `chmod u+xs`、default ACL、`chown`/`chgrp` の
  `--from`（フィルタ・positional 残置）vs `--reference`（positional 不在）区分、`ln` のシンボリック（`-s`/`--symbolic`、
  リンク親基準）vs ハードリンク（CWD 基準）相対パス解決、`tar` の第1トークン（およびダッシュ始まりトークン）限定モード解析）が
  回帰テストとして緑である。

### F-004: 既存挙動の保存（挙動を変えないリファクタ）

本タスクはリファクタであり、観測可能な挙動を一切変えない。

**Acceptance Criteria**:
- **AC-09**: リファクタ前後で 0142 の全テスト（`security/destination_zoning_test.go`・`security/operand_path_resolver_test.go`・
  関連統合テスト）が**変更なし**で緑である。テストの期待値変更を伴う場合は挙動変更とみなし、本タスクの不適合とする。
- **AC-10**: 26 対象コマンド×代表フラグの表駆動テストで、各入力に対する `LocationResult`（`Applies`/`Recognized`/`Level`/
  `Operands`（Index/Raw/Resolved/Zone/Role/MatchedCritical/Trusted）/`ReasonCodes`）がリファクタ前と**同一**である
  （挙動差分ゼロ）。

### F-005: fail-closed 保証の保存

0142 の安全保証である fail-closed `Recognized` contract を保存する。

**Acceptance Criteria**:
- **AC-11**: 未知/曖昧形・解決不能オペランド・値フラグの値欠落・不完全形（必須 positional 欠落）は、リファクタ後も
  `Recognized=false` となり High 下限へ倒れる（fail-closed）。トークンを暗黙に捨てる経路が存在しないことを、AC-04 の表明と
  併せて担保する。

## 4. 非機能要件

- **NF-001**: `make fmt`・`make test`・`make lint` が成功する。`./internal/runner/...` がコンパイル・緑であること。
- **NF-002**: 保守性: 新規ファイル操作コマンドの追加が、宣言的仕様エントリ 1 つ＋（必要なら）薄いロール割当の追加で完結する
  （コマンドごとの独自パーサ実装を要さない）ことを、追加例で示す。抽出層の引数解析の重複が削減される。
- **NF-003**: 決定性・read-only を維持する（live identity 不参照・副作用なし。0142 NF-003 を継承）。リファクタは解析の構造のみを
  変え、解決・区分判定・identity 注入の純粋性を変えない。

## 5. スコープ外の根拠

- **挙動変更を伴う改善は本タスクで行わない**: 例えば「より多くのフラグを精密解析して過剰分類（fail-closed）を減らす」改善は
  価値があるが、挙動保存（F-004）と両立しないため別タスクとする。本タスクの完了基準は「同一挙動・構造のみ刷新」である。
- **判断軸2 の意味論・0143 の監査出力は対象外**: レベル規則・区分・下限・コマンド別特則（0142）と logger 出力・文書（0143）は
  既存の所掌に従い、本タスクは抽出層の内部実装にのみ介入する。

## 6. 成功基準（要約）

- AC-01〜AC-11 が test/static で緑。
- 0142 の全既存テストが**無改変**で緑（挙動保存の最重要証拠）。
- 単一 getopt パーサ＋宣言的仕様＋完全性メタテストにより、エイリアス／optional-arg／fail-closed 一貫性／未分類 path 値の
  各クラスが構造的に再発不能になっている。
