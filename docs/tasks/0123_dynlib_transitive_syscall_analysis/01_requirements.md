# 動的リンクライブラリの再帰的システムコール解析 要件定義書

## 1. 概要

### 1.1 背景

タスク 0070/0072（ELF 機械語解析）および 0082（動的リンクライブラリのシンボル解析）により、`record`
コマンドは以下の手段で実行ファイルのシステムコール利用を検出できるようになっている。

| 手段 | 対象 | タスク |
|------|------|--------|
| `.dynsym` UNDEF シンボル解析 | 動的リンクバイナリ | 0069 |
| 機械語 `syscall` 命令スキャン | 全 ELF（静的・動的） | 0070/0072 |
| SOName ベース既知ネットワークライブラリ照合 | DT_NEEDED ライブラリ | 0082 |

しかし、実行ファイルが動的リンクしている**アプリケーションレベルライブラリ**（例:
`libssl.so`, `libcurl.so`, `libruby.so` 等）がシステムコールを呼び出している場合、現行の解析では
以下のケースを見逃す可能性がある。

**見逃しが発生する典型例:**

```
実行ファイル → libfoo.so → libc.so → socket() [syscall]
```

- 実行ファイルの `.dynsym` に `socket` は現れない（libfoo.so が隠蔽）
- `libfoo.so` は SOName ベース既知リストに未登録
- 実行ファイル自体に `syscall` 命令なし（Go バイナリでもない）

この構造は C/C++ で書かれた実行ファイルが専用のミドルウェア共有ライブラリに実装を
委譲している場合によく見られる。

### 1.2 目的

`record` コマンドが実行ファイルを解析する際に、動的リンクされているライブラリ自体も
システムコール解析の対象とし、以下を実現する。

1. 実行ファイルが libc 等のシステムコールラッパを経由して直接呼び出しているシステムコールを
   検出する（**現行と同じ**）
2. 実行ファイルが**システムコールラッパ以外のライブラリ**を動的リンクしている場合、そのライブラリが
   libc 経由または直接呼び出しているシステムコールを検出する（**新規**）
3. ライブラリが他のライブラリをリンクしている場合、再帰的にシステムコール検出の対象とする（**新規**）

本機能により、SOName ベース既知ネットワークライブラリリスト（タスク 0082）に未登録の
ライブラリ経由のネットワーク利用も検出できるようになる。

### 1.3 スコープ

- **対象プラットフォーム**: ELF (Linux) のみ（x86\_64、arm64）
- **対象**: 動的リンクされた ELF 実行ファイル（DT\_NEEDED エントリを持つもの）
- **分析単位**: ライブラリ単位（関数・呼び出しグラフ単位の解析は行わない）
- **再帰深度**: 現行の `MaxRecursionDepth = 20` を流用
- **非対象**: macOS Mach-O バイナリ、Windows PE バイナリ、スクリプトファイル
- **非対象**: 動的ロード（`dlopen`）で実行時にロードされるライブラリ（静的解析では追跡不可）

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| システムコールラッパライブラリ | libc、ld-linux 等、OS ABI レイヤーとしてシステムコールを直接ラップするライブラリ。本機能では解析対象から除外する |
| アプリケーションライブラリ | libssl、libcurl 等、システムコールラッパ以外のライブラリ。本機能の解析対象 |
| DynLibDeps | `record` コマンドが記録する動的リンクライブラリ依存情報（DT\_NEEDED + 推移的依存） |
| 解析済みライブラリキャッシュ | `record` 実行時に同じライブラリを重複解析しないためのセッション内キャッシュ |
| LibraryAnalysisResult | ライブラリ単位のシステムコール解析結果（新規フィールド） |

## 3. 機能要件

### 3.1 システムコールラッパライブラリの判定

#### FR-3.1.1: システムコールラッパ SOName プレフィックスリスト

以下の SOName プレフィックスをシステムコールラッパとして定義し、`record` 時のライブラリ
解析対象から除外する。

照合方式: SOName が `<prefix>` で始まり、その直後が `.`, `-`, 数字のいずれかである場合のみ一致。
（タスク 0082 の既知ネットワークライブラリリストと同じ方式）

| SOName プレフィックス | 対応ライブラリ | 除外理由 |
|---|---|---|
| `libc` | glibc（libc.so.6 等） | システムコールを直接ラップする OS ABI 層 |
| `libpthread` | POSIX スレッドライブラリ（glibc 組み込み可） | システムコールラッパ（futex, clone 等）を直接含む |
| `libdl` | 動的リンクライブラリ（glibc 組み込み可） | システムコールラッパを直接含む |
| `librt` | POSIX リアルタイム拡張（glibc 組み込み可） | システムコールラッパを直接含む |
| `libgcc_s` | GCC サポートライブラリ | 例外処理等の低レベルランタイム |
| `ld-linux` | ELF 動的リンカ（x86\_64） | 動的リンカ本体、解析対象外 |
| `ld-linux-x86-64` | ELF 動的リンカ（x86\_64 explicit） | 同上 |
| `ld-linux-aarch64` | ELF 動的リンカ（arm64） | 同上 |
| `linux-vdso` | Linux vDSO | カーネルが提供する仮想 DSO、ファイルとして存在しない |

> **注記**: `libm`（数学ライブラリ）と `libstdc++`（C++ 標準ライブラリ）はシステムコールを
> 直接含む可能性がある（例: libstdc++ の例外 / RTTI）が、ネットワーク系システムコールを
> 含む可能性は低い。本バージョンでは除外リストに含めない（解析対象のままとする）。
> 解析結果が実態に合わない場合は後続タスクで除外対象に追加する。

#### FR-3.1.2: VDSO の除外

`linux-vdso.so.1` 等の仮想 DSO はファイルとして存在しないため、ライブラリ解析の前段で
除外する（現行の `knownVDSOs` マップを流用）。

### 3.2 ライブラリ解析

#### FR-3.2.1: 解析対象ライブラリの収集

`record` 実行時に解析対象の実行ファイルの DynLibDeps（推移的依存含む）から、以下の
条件を満たすライブラリを解析対象として収集する。

収集条件:
1. `knownVDSOs` に含まれないこと（FR-3.1.2）
2. SOName が FR-3.1.1 のシステムコールラッパリストに一致しないこと
3. ファイルが実際に存在し、読み取り可能であること
4. 解析済みライブラリキャッシュに存在しないこと（重複解析防止）

再帰深度は `MaxRecursionDepth = 20` を上限とする（現行と同じ）。

#### FR-3.2.2: ライブラリの機械語解析

収集されたアプリケーションライブラリに対して、実行ファイルと同じ機械語 `syscall`
命令スキャン（タスク 0070/0072）を適用する。

- x86\_64: `syscall` 命令（オペコード `0F 05`）を検出し、`eax`/`rax` の即値から
  システムコール番号を抽出（タスク 0070 と同じアルゴリズム）
- arm64: `svc #0` 命令を検出し、`x8` レジスタの即値からシステムコール番号を抽出
  （タスク 0072 と同じアルゴリズム）
- 番号不明の場合: 既存と同様に `unknown:indirect_setting` として記録

#### FR-3.2.3: ライブラリの `.dynsym` UNDEF シンボル解析

収集されたアプリケーションライブラリに対して、実行ファイルと同じ `.dynsym` UNDEF
シンボル解析（タスク 0069）を適用する。

- ライブラリが `socket`, `connect`, `bind` 等の既知ネットワーク libc シンボルを
  UNDEF（= libc から import）として持つ場合、ネットワーク利用ありとみなす

#### FR-3.2.4: 解析結果の記録

ライブラリの解析結果（検出されたシステムコール番号・シンボル）を
`fileanalysis.Record` の新フィールド `LibraryAnalysis []LibraryAnalysisEntry` に記録する。

```
LibraryAnalysisEntry {
    SOName  string              // ライブラリの SOName（例: "libfoo.so.1"）
    Path    string              // 解析したライブラリの絶対パス
    // ライブラリ自身の機械語解析・dynsym 解析結果
    SyscallAnalysis   *SyscallAnalysisData    // 機械語解析結果（nil = 解析なし or 結果なし）
    SymbolAnalysis    *SymbolAnalysisData      // .dynsym UNDEF 解析結果（nil = 解析なし）
}
```

#### FR-3.2.5: runner によるリスク判定への反映

`runner` 実行時に `LibraryAnalysis` フィールドを参照し、いずれかのライブラリで
ネットワーク系システムコール（`socket`, `connect` 等）または既知ネットワークシンボルが
検出されている場合、当該コマンドをネットワーク利用ありと判定する。

判定基準は実行ファイル自身の解析結果に適用している基準と同じとする。

### 3.3 キャッシュ・パフォーマンス

#### FR-3.3.1: セッション内解析済みキャッシュ

同一の `record` 実行内で同じライブラリファイル（パスで同定）を複数の実行ファイルから
参照する場合、2 回目以降はキャッシュを返す。

`record` の複数コマンド解析時に共有ライブラリ（libc や libssl 等）が重複解析されるのを
防ぐ。

#### FR-3.3.2: ライブラリの最大解析ファイルサイズ

実行ファイルと同じ `maxFileSize = 1 GB` を上限とする。超過した場合は解析をスキップし、
警告を `AnalysisWarnings` に記録する。

### 3.4 エラー処理

#### FR-3.4.1: ライブラリが見つからない場合

ライブラリファイルが存在しない・読み取り不可の場合は `AnalysisWarnings` に警告を
追記し、処理を継続する（実行ファイルの解析を中断しない）。

#### FR-3.4.2: 機械語解析エラー

ライブラリ機械語解析中にエラーが発生した場合は `AnalysisWarnings` に追記し、
そのライブラリの解析結果は「解析不能」として記録する。

---

## 4. 非機能要件

### 4.1 パフォーマンス

- 解析時間の大幅な増加を避けるため、FR-3.3.1 のキャッシュを必ず適用する
- 1 つのライブラリの解析時間は現行の実行ファイル解析と同等とする

### 4.2 スキーマバージョン

`CurrentSchemaVersion` を 19 → 20 に更新する。

## 5. 受け入れ基準

| ID | 基準 |
|----|------|
| AC-1 | `libfoo.so` が `socket()` を UNDEF に持つ場合、`libfoo.so` にリンクした実行ファイルを `record` すると `LibraryAnalysis` に `libfoo.so` のシンボル解析結果が記録される |
| AC-2 | `libbar.so` が直接 `syscall` 命令（socket syscall 番号 = 41 on x86\_64）を含む場合、`libbar.so` にリンクした実行ファイルを `record` すると `LibraryAnalysis` にシステムコール解析結果が記録される |
| AC-3 | `libc.so` にリンクしているだけで `libc.so` 自体は `LibraryAnalysis` に記録されない（システムコールラッパ除外） |
| AC-4 | `ld-linux-x86-64.so.2` は `LibraryAnalysis` に記録されない（動的リンカ除外） |
| AC-5 | `LibraryAnalysis` にネットワーク系システムコールまたはシンボルが記録されたコマンドは、`runner` によりネットワーク利用ありと判定される |
| AC-6 | 同一ライブラリが複数の実行ファイルの依存グラフに現れる場合、セッション内で1回だけ解析される（キャッシュ確認） |
| AC-7 | 存在しないライブラリパスに対しては `AnalysisWarnings` に警告が追加され、処理が継続される |
| AC-8 | 既存の `.dynsym` UNDEF 解析・機械語 syscall 解析の動作が変わらないこと（回帰テスト） |

## 6. 実現可能性・False Positive リスク評価

### 6.1 技術的実現可能性

本機能は既存の解析エンジン（機械語 syscall スキャン・`.dynsym` UNDEF 解析）を
ライブラリ単位に拡張するものであり、新規アルゴリズムは不要である。DynLibDeps の
推移的依存情報も既に取得・記録済みのため、解析対象の収集も容易である。

**実現可能性: 高**

### 6.2 False Positive（過検知）リスク

本機能の最大の課題は**ライブラリ単位での解析粒度**に起因する false positive である。

#### 6.2.1 False Positive の発生メカニズム

ライブラリには、**リンクした実行ファイルがすべての機能を使うとは限らない**という
本質的な問題がある。例えば:

```
xmlparser バイナリ → libxml2.so
libxml2.so には:
  - XML パース機能（ネットワーク不使用）
  - HTTP フェッチ機能（socket syscall を含む） ← 実際には呼ばれない
```

`libxml2.so` を解析すると HTTP フェッチ用の `socket()` シンボルまたは
`syscall` 命令が検出されるため、「ネットワーク利用あり」と誤検知される。

> **注記**: タスク 0082 は、まさにこのケース（libxml2 の全 UNDEF シンボル解析による
> false positive）を理由に「全 UNDEF シンボル解析」アプローチを**却下**し、
> SOName ベース既知リストアプローチを採用した経緯がある。
> 本機能はより詳細な機械語解析も含むが、関数単位の解析を行わない限り
> 同じ false positive リスクを抱える。

#### 6.2.2 ライブラリ別 False Positive リスク分類

| ライブラリ種別 | 例 | False Positive リスク | 理由 |
|---|---|---|---|
| ネットワーク専用ライブラリ | libcurl, libssl, libzmq | **低** | 目的がネットワーク通信であるため、リンクすれば実際に使用している可能性が高い |
| 言語ランタイムライブラリ | libruby, libpython | **低〜中** | ランタイム自体はネットワーク可能だが、スクリプト次第（すでに SOName 既知リストで対処済み） |
| 多目的ミドルウェア | libxml2, libglib-2.0 | **高** | ネットワーク機能を含むが、使用しない機能も多い |
| セキュリティ・暗号 | libcrypto (OpenSSL) | **中** | 暗号処理のみの利用（libssl なし）でも socket を含む可能性あり |
| データベースクライアント | libpq (PostgreSQL), libmysqlclient | **低〜中** | 通常はネットワーク接続を伴うが、ローカルソケット利用のケースもある |
| 圧縮・変換 | libz, libpng, libjpeg | **低** | ネットワーク系 syscall を含まないため false positive が発生しにくい |

#### 6.2.3 False Positive リスクの定量的推定

Linux 環境で使われる共有ライブラリのうち、`socket()` 等のネットワーク libc シンボルを
UNDEF に持つライブラリの割合を `/usr/lib` の典型的なパッケージで調査すると:

- `libcurl.so.4`: `socket` UNDEF あり → 正検知
- `libssl.so.3`: `socket` UNDEF あり（TLS 前提） → 正検知
- `libxml2.so.2`: `socket` UNDEF あり（HTTP フェッチ機能） → **false positive 発生しうる**
- `libglib-2.0.so.0`: `socket` UNDEF あり（GSocket） → **false positive 発生しうる**
- `libsqlite3.so.0`: `socket` UNDEF なし → 正検知（非検出）
- `libz.so.1`: `socket` UNDEF なし → 正検知（非検出）

特定の多目的ライブラリを持つ実行ファイルでは false positive が**高確率**で発生する。
ただし false negative（ネットワーク利用を見逃すこと）よりも false positive の方が
セキュリティ上は保守的（安全側）の誤りである。

#### 6.2.4 False Positive に対するリスク軽減策

| 軽減策 | 説明 | 実装コスト |
|---|---|---|
| **除外リスト（本機能採用）** | 解析対象から特定ライブラリを除外するリスト（FR-3.1.1 のラッパーリストとは別に、既知 false positive 生成ライブラリを除外） | 低（リスト管理のみ） |
| **SOName 既知リストとの統合** | SOName 既知ネットワークライブラリリスト（タスク 0082）に登録済みのものは既知リスト由来で記録し、未登録のもののみ機械語解析する | 中 |
| **関数レベル到達可能性解析** | 実行ファイルのエントリポイントから呼び出しグラフを辿り、実際に到達可能な関数のみを解析 | **非常に高**（本機能では非対象） |
| **`--allow-library-network` フラグ** | 特定ライブラリのネットワーク検出を明示的に許容するオプション | 低（設定変更のみ） |

**本機能での採用策**: 除外リストによる既知 false positive ライブラリの管理と、
`AnalysisWarnings` への記録による利用者への通知。完全な false positive 排除は
関数レベル解析なしには達成できないため、本機能の限界として文書化する。

### 6.3 False Negative（見逃し）リスク

| シナリオ | リスク | 理由 |
|---|---|---|
| `dlopen()` による実行時ロード | **高** | 静的解析では追跡不可。別途 `dlopen` 検出（タスク 0069 の `DynamicLoadSymbols`）でカバー |
| 難読化・パック済みバイナリ | **中** | 解析に使えるセクションが存在しない可能性 |
| JIT コンパイル（V8, JVM 等） | **高** | 実行時生成コードは静的解析不可 |

### 6.4 まとめ

| 項目 | 評価 |
|---|---|
| 技術的実現可能性 | 高（既存エンジンの再利用） |
| False Positive リスク | **中〜高**（多目的ミドルウェアライブラリを依存に持つ実行ファイルで発生）|
| False Negative リスク | 中（dlopen・JIT は静的解析の限界） |
| 推奨対応 | 機能を実装しつつ、false positive が発生しやすいライブラリの除外リストを整備し、ユーザーが許容リストを設定できる仕組みを提供する |

## 7. 制約事項・設計上の限界

1. **関数単位の解析は行わない**: 本機能はライブラリ全体を解析対象とするため、
   実際の呼び出しパスで到達しないシステムコールも検出される可能性がある（6.2 節参照）。
   これは意図した設計上の制約であり、関数レベル到達可能性解析は将来の検討課題とする。

2. **`dlopen` 経由のライブラリは非対象**: 実行時に動的にロードされるライブラリは
   静的解析の限界により追跡できない。タスク 0069 の `DynamicLoadSymbols` 機能で
   `dlopen` の使用有無を検出できるが、ロードされるライブラリの内容は解析できない。

3. **クロスアーキテクチャ解析は行わない**: 解析対象ライブラリのアーキテクチャは
   実行ファイルと同じものに限定する。異なるアーキテクチャのライブラリが発見された場合は
   `AnalysisWarnings` に記録し、スキップする。

## 8. 関数単位解析粒度の実現可能性評価

本機能（ライブラリ単位解析）の代替アプローチとして、関数単位（呼び出しグラフ）解析を
採用した場合の実現可能性を評価する。

### 8.1 解析アルゴリズムの概要

関数単位解析の手順は以下のとおり。

1. **インポート関数の特定**: 実行ファイルの `.dynsym` UNDEF エントリと `.rela.plt` から、
   ライブラリから import している関数名の一覧を取得する。
2. **ライブラリ内関数の特定**: ライブラリの `.symtab` または `.dynsym` から、
   手順 1 で取得した関数名のアドレスと範囲を求める。
3. **呼び出しグラフの構築**: 関数本体の機械語を前方スキャンし、CALL 命令（x86\_64）/
   BL 命令（arm64）のターゲットアドレスを解決して呼び出し先関数を特定、BFS で再帰的に辿る。
4. **到達可能な syscall の収集**: 到達した関数内で `syscall`/`svc` 命令または
   ネットワーク libc シンボルの UNDEF エントリを検出する。

### 8.2 既存実装の活用可能性

コードベースには既に以下の実装が存在し、関数単位解析の基盤として再利用できる。

| 機能 | ファイル | 関数単位解析での用途 |
|------|---------|---------------------|
| `GetCallTarget()` | `x86_decoder.go`, `arm64_decoder.go` | 直接 CALL 命令のターゲットアドレス解決 |
| `IsControlFlowInstruction()` | 同上 | 関数境界（`ret`）の検出 |
| `AnalyzeSyscallsInRange()` | `syscall_analyzer.go` | 指定アドレス範囲内の syscall 検出 |
| `findFuncPLTAddr()` | `plt_analyzer.go` | ライブラリの PLT 経由 libc 呼び出しの特定 |

`AnalyzeSyscallsInRange()` は既に「コードの任意範囲」を解析できるため、
関数境界を特定できれば即座に適用可能である。

### 8.3 技術的障壁

#### 8.3.1 ストリップ済みライブラリ（最大の障壁）

関数単位解析の最大の障壁は、**プロダクション環境のライブラリがシンボルテーブルを
ストリップされている場合がある**点である。

| シンボル情報 | 内容 | ストリップ影響 |
|---|---|---|
| `.symtab` | 全シンボル（内部関数含む）の名前・アドレス・サイズ | **ストリップで消える** |
| `.dynsym` | エクスポートされたシンボルのみ（常に存在） | ストリップ後も残る |

`.symtab` がない場合、以下の問題が生じる。

- **関数境界が不明**: 関数の開始アドレスは `.dynsym` のエクスポートシンボルから取得できるが、
  内部関数（プライベート関数）のアドレスはわからない。呼び出しグラフの葉ノードが内部関数で
  ある場合、その先の解析ができない。
- **サイズが不明**: ELF の `STT_FUNC` シンボルはサイズ（`st_size`）を持つが、
  `.symtab` がない場合は「次のシンボルとのアドレス差分」を関数サイズと推定するしかなく、
  誤差が生じる。

**Ubuntu の典型的なライブラリ:**
- `libssl.so.3`, `libcrypto.so.3`, `libxml2.so.2`: `.symtab` 存在（非ストリップ）
- 多くのサードパーティ製パッケージやクロスコンパイル成果物: `.symtab` なし（ストリップ済み）

#### 8.3.2 間接呼び出し（関数ポインタ・仮想ディスパッチ）

CALL 命令のターゲットがレジスタ（`call rax`）またはメモリ（`call [rip+offset]`）の場合、
`GetCallTarget()` はターゲットアドレスを解決できない。

典型的な発生ケース:
- C の関数ポインタ（コールバック）: `call rax`, `blr x8`
- C++ の仮想関数ディスパッチ（vtable 経由）: `call [rax+0x18]`
- GOT 経由のグローバル関数ポインタ

典型的な C ライブラリにおける間接呼び出しの割合は、コンパイラの最適化や
コーディングスタイルによるが、**15〜40% 程度**が間接呼び出しになる場合がある。
これらは静的解析では追跡不可能であり、到達可能性の判定に穴を生じさせる。

> **注記**: 間接呼び出しが追跡できない場合、保守的に「到達可能」とみなして
> false positive を増やすか、「到達不可能」とみなして false negative を増やすか
> を選択する必要がある。

#### 8.3.3 テールコール（tail call）

一部のコンパイラ最適化では、CALL + RET を `JMP`（テールコール）に変換する。
現行の `IsControlFlowInstruction()` は JMP を制御フロー命令として検出するが、
テールコールを「関数終端」と判定するか「別関数への遷移」と判定するかは文脈依存であり、
誤判定が生じる可能性がある。

#### 8.3.4 呼び出しグラフの爆発的拡大

大規模ライブラリ（例: `libssl.so`, `libglib-2.0.so`）では、エントリ関数から
到達可能な関数数が数百〜数千になる場合がある。深さ制限（例: 深さ 10）を設けると
false negative が増加し、制限しないと解析時間・メモリが爆発する。

### 8.4 実現可能性の総合評価

| 評価軸 | ライブラリ単位解析（本機能） | 関数単位解析（代替案） |
|---|---|---|
| 実装コスト | 低（既存エンジン再利用） | **高**（新規呼び出しグラフ構築が必要） |
| 非ストリップライブラリでの false positive 削減 | なし | **大きい**（libxml2 ケース等） |
| ストリップ済みライブラリへの対応 | 完全対応 | **部分的**（内部関数境界が不明） |
| 間接呼び出しのある関数の精度 | 対象外 | 穴が生じる（保守的か非保守的のどちらか） |
| 解析時間 | 低（全体スキャン） | 中〜高（グラフ探索） |
| 実装の複雑さ | 低 | **高**（500〜1,000 行規模の新規コード） |

**結論**: 関数単位解析は技術的には実現可能であるが、以下の条件に強く依存する。

- `.symtab` が存在するライブラリ（非ストリップ）に限り、高精度な false positive 削減が期待できる
- `.symtab` がないライブラリではライブラリ単位解析にフォールバックが必要
- 間接呼び出しが多いライブラリ（C++、コールバック多用）では依然として false positive が残る

段階的実装アプローチとして、「`.symtab` ありのライブラリのみ関数単位解析、
なしはライブラリ単位解析」とするハイブリッド方式が現実的だが、
その場合でも呼び出しグラフ構築（新規コンポーネント）の実装コストは変わらない。

**優先度**: 本機能（ライブラリ単位解析）を先に実装し、false positive の発生状況を
実際のユースケースで観測した上で、関数単位解析への拡張を将来タスクとして検討する。
