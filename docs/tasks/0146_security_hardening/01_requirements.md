## 1. 背景と目的

### 背景

`go-safe-cmd-runner`（ブランチ `main`, commit `a5b82894`）に対するレビューベースの
セキュリティ監査結果を [`00_security_risk_report.md`](00_security_risk_report.md) にまとめた。
本アプリケーションは「非特権ユーザーに特権操作を安全に委譲する」ことを目的とした
多層防御設計であり、一般的な攻撃ベクトルの多くは既に塞がれている。一方で、堅牢な基盤の
上に **残余リスク／堅牢化の余地** が確認された。本タスクはそれらを是正する。

レポートの検出事項（重要度付き）:

| レポートID | 重要度 | 概要 | 該当箇所 | 本書での対応要件 |
|---|---|---|---|---|
| F-1 | **High** | run-as ユーザー切替時に補助グループ（supplementary groups）を破棄していない | `internal/runner/base/privilege/unix.go` | F-001 |
| F-2 | Medium | ユーザー/グループ切替が `seteuid/setegid` のみで saved-set-uid を明示的に落としていない | `internal/runner/base/privilege/unix.go` | F-002 |
| F-3 | Medium | 機微情報マスキングがキー名パターン依存で、値中の生の秘密値を取りこぼす可能性 | `internal/redaction/sensitive_patterns.go` | F-003 |
| F-4 | Low | dry-run 時に検証失敗した設定ファイルを `os.ReadFile` で読み直し、未検証内容を解析対象にする | `internal/verification/manager.go` | F-004 |
| F-5 | Low | `record` の TOCTOU/ディレクトリ権限チェックが警告のみで、信頼の起点であるハッシュDB生成を止めない | `cmd/record/main.go` | F-005 |
| F-6 | Info | 128MB のファイルサイズ上限により大容量バイナリの検証・解析が不可（可用性） | `internal/safefileio/safe_file.go`, `filevalidator` | F-006 |
| F-7 | Info | non-Linux（openat2 非対応）環境で TOCTOU 残余ウィンドウが存在 | `internal/safefileio/safe_file.go` | F-007 |

### 目的

1. setuid-root 配備時の **権限分離を完全化** する（F-1 → F-001、F-2 → F-002）。
   run-as で一般ユーザーに切り替えたコマンドが、起動元（root 等）の補助グループや
   saved-set-uid を引き継がないことを保証する。
2. 機微情報の **外部流出面（特に Slack 通知・ログ）を縮小** する（F-3 → F-003）。
   キー名を伴わない秘密値も値ベースで検出・マスクする。
3. **信頼の起点（record によるハッシュDB生成）と dry-run の安全側デフォルト** を固める
   （F-4 → F-004、F-5 → F-005）。
4. 可用性上・環境上の **前提と限界をドキュメントで明確化** し、必要に応じて設定可能化する
   （F-6 → F-006、F-7 → F-007）。

### スコープ外

- リスク判定アルゴリズム（`risk.StandardEvaluator` 等）の変更。本タスクは権限・マスキング・
  信頼の起点の堅牢化に限定する。
- バイナリ静的解析アルゴリズム自体の変更。
- 高エントロピー文字列ヒューリスティックの **必須化**。F-003 では既知フォーマットの
  値ベース検出を必須とし、エントロピーヒューリスティックは誤検知リスクを考慮して
  設計判断（`02_architecture.md`）に委ねる任意事項とする。
- F-6 のファイルサイズ上限の **具体的な閾値変更**。本タスクでは前提の明文化と、
  設定可能化 or ハッシュ/解析の上限分離の可否検討までとする。

### 前提・依存

- F-001 / F-002 の実害範囲は配備形態（setuid-root か native-root か、`run_as` の使用有無）に
  依存する。native root 実行・`run_as` 未使用の経路では既存挙動を回帰させないこと。
- 補助グループ・saved-set-uid のアトミック設定について、レポートは
  `os/exec` の `SysProcAttr.Credential{Uid, Gid, Groups, NoSetGroups:false}` への移行を推奨している。
  採否と適用範囲（コマンド実行のみカーネル側アトミック設定に移し、ファイル検証等の
  親プロセス内特権処理は現行 `WithPrivileges` を維持する等）は `02_architecture.md` で確定する。
  本書は **振る舞い（WHAT）** を定義し、実現機構（HOW）は設計文書に委ねる。
- **AC 番号について**: 要件プロセスの規則（欠番は許容、既存番号の振り直し禁止）に従う。
  本書の AC は AC-01〜AC-25。

## 2. 機能要件

### F-001: run-as ユーザー切替で補助グループを対象ユーザーのものに再設定する（レポート F-1 / High）

**現状の問題**
`UnixPrivilegeManager.changeUserGroupInternal` は対象ユーザー/グループへの切替を
`Setegid` と `Seteuid` のみで行い、`setgroups(2)` を呼ばない。コードベース全体に
`Setgroups` の呼び出しは存在しない。setuid-root 配備時、プロセス起動時の補助グループは
起動元（root）の補助グループ集合（例: `wheel`, `sudo`, `docker`, `adm`）であり、`run_as_user`
で一般ユーザーに切り替えても補助グループは root のものが残る。`docker` グループが残れば
実質 root 相当、group 読み取り可能な機微ファイルへのアクセス等、権限分離の前提が崩れる。

**要件**
run-as によるユーザー切替を行う場合、子プロセスの補助グループ集合を **対象ユーザーの
初期補助グループ集合**（`getgrouplist` 相当）に再設定し、起動元プロセスの補助グループを
引き継がせない。補助グループの決定・設定は uid/gid の切替と整合していなければならない。

**Acceptance Criteria**:
- **AC-01**: `run_as` によるユーザー切替で起動されたコマンドの補助グループ集合が、対象ユーザーの
  初期補助グループ集合（`getgrouplist` 相当）と一致し、起動元プロセス（root 等）の補助グループを
  1 つも引き継がない。
- **AC-02**: 補助グループの決定または設定に失敗した場合、コマンドを実行せずフェイルクローズ
  （エラー返却・非ゼロ終了）する。root の補助グループを保持したまま実行を継続しない。
- **AC-03**: 補助グループの再設定は uid/gid の切替と整合し、切替後の子プロセスの実効・実
  uid / gid / 補助グループが対象ユーザーの identity と一致することを検証可能な形で確認できる。
- **AC-04**: `run_as` を使用しない経路（native root 実行、ユーザー/グループ変更なし、dry-run）では
  補助グループ操作を行わず、既存挙動を回帰させない。

### F-002: ユーザー切替で real/effective/saved を一括設定し saved-set-uid を残さない（レポート F-2 / Medium）

**現状の問題**
特権実行は「親プロセスの effective UID を一時変更 → コマンドを同期実行 → 復元」方式で、
切替に `Seteuid`/`Setegid` を用い real / saved-set-uid を明示的に変更しない。setuid-root 配備時、
`run_as` 実行中の親プロセスは `euid=target, ruid=originalUser, suid=0` となる瞬間があり、
saved-set-uid=0 を保持したまま処理が進む区間が存在する。`restoreUserGroupInternal` →
`restorePrivileges` の二段構えと emergency shutdown を組み合わせた復元ロジックは複雑で、
将来の改修時に権限リークを生みやすい構造である。

**要件**
run-as のためのユーザー/グループ切替を行う区間で、saved-set-uid=0 を保持したまま
コマンドを実行しない。次のいずれかで実現する（機構選定は `02_architecture.md`）:
- コマンド実行をカーネル側アトミック設定（`SysProcAttr.Credential`）に移し、親プロセスを
  一時 seteuid する区間そのものを排除する。ファイル検証等の親プロセス内特権処理は
  現行 `WithPrivileges` を維持してよい。
- 現行方式を維持する場合は、切替時に real/effective/saved を一括設定し、復元後の不変条件
  チェックに saved-set-uid の検証を加える。

**Acceptance Criteria**:
- **AC-05**: run-as コマンド実行区間で親プロセスの saved-set-uid が 0（＝起動元 root）のまま
  子プロセスの実行が行われない。実現機構に応じて、(a) 一時 seteuid 区間が存在しない、または
  (b) 切替時に real/effective/saved が対象値に一括設定される、のいずれかを満たす。
- **AC-06**: 権限復元後の不変条件チェックが saved-set-uid（suid）も検証対象に含める。復元後の
  euid/egid/suid のいずれかが原状と異なる場合は既存の emergency shutdown（即時プロセス停止）を行う。
- **AC-07**: 既存の EUID==UID / EGID==GID 不変条件チェックおよび復元失敗時の即時停止は維持される
  （回帰なし）。

### F-003: 機微情報を値ベースでも検出・マスクし外部流出面を縮小する（レポート F-3 / Medium）

**現状の問題**
`redaction` パッケージの機微判定は主にキー名／変数名に対する正規表現
（`password|token|secret|key|api_key` 等）で行われ、`IsSensitiveValue` も同じキー向けパターンを
流用している。任意形式の秘密値（高エントロピー文字列、`AKIA...` 形式、JWT、PEM/private key
ブロック等）を値そのものから検出する仕組みは限定的である。コマンド引数・stdout/stderr・
環境変数値に秘密情報が「キー名を伴わず」現れた場合、ログや **Slack 通知** に平文で流出し得る。
Slack はホスト検証（`slack_allowed_host`）で送信先を制限するが、送信「内容」のマスキングは
上記パターンに依存する。

**要件**
キー名を伴わない秘密値を **値そのもの** から検出しマスクする値ベース検出器を追加する。
少なくとも既知フォーマットを対象とし、外部送信経路（Slack）に確実に適用する。

**Acceptance Criteria**:
- **AC-08**: 既知フォーマットの秘密値がキー名を伴わなくても値として検出・マスクされる。
  対象は少なくとも次を含む: クラウドプロバイダの資格情報プレフィックス（AWS `AKIA`/`ASIA` 等、
  GitHub `ghp_`/`gho_`/`ghs_` 等、Slack `xox` 系、GCP サービスアカウント）、PEM/private key ブロック、
  `Bearer <token>`、URL 埋め込み credential（`scheme://user:pass@host`）。
- **AC-09**: 値ベース検出がコマンド引数・stdout/stderr・環境変数値に適用される。少なくとも
  外部送信経路（Slack 通知）へ載る内容には必ず適用される。
- **AC-10**: Slack 送信ペイロードは、送信前に値ベースマスキングを必ず通すか、許可リスト化された
  フィールドのみを送るホワイトリスト方式とし、自由記述本文に未マスクの生コマンド出力を載せない。
- **AC-11**: `--show-sensitive` を明示指定した場合のみ平文化される既存挙動を維持し、既定はマスクとする。
- **AC-12**: 「Slack にコマンド出力を載せる設定は避けるべき」旨と、値ベースマスキングの適用範囲・限界が
  ユーザー向けドキュメントに明記される。

### F-004: dry-run で未検証内容を明示区別し、CI 向けに hard fail 可能にする（レポート F-4 / Low）

**現状の問題**
`readAndVerifyFileWithReadFallback` は dry-run モードで検証に失敗しても `os.ReadFile` で内容を
読み直して処理を継続する（失敗は ResultCollector に記録）。未検証（改ざんされ得る）内容が
変数展開・リスク評価・デバッグ出力に流れ込み、「dry-run では問題なし」と誤認させる（誤った安心を
与える）リスクがある。

**要件**
dry-run のフォールバック読み込みで得た未検証内容を出力上で明確に区別し、CI 用途では未検証を
hard fail にできるようにする。

**Acceptance Criteria**:
- **AC-13**: dry-run で検証に失敗しフォールバック読み込みした内容は「UNVERIFIED」として dry-run 出力上で
  明確に区別表示される（検証済み内容と混同されない）。
- **AC-14**: 未検証を hard fail（非ゼロ終了）にするオプションが提供され、その既定挙動と CI 用途での
  推奨運用がドキュメント化される。
- **AC-15**: 上記変更後も、正常系（検証成功）の dry-run 出力および非 dry-run 実行の挙動を回帰させない。

### F-005: record のディレクトリ権限違反をフェイルクローズにする（レポート F-5 / Low）

**現状の問題**
`cmd/record/main.go` は TOCTOU/ディレクトリ権限チェック（`RunTOCTOUPermissionCheck`）を実行するが、
結果は警告ログのみで違反があっても処理を続行する。ハッシュディレクトリは `os.MkdirAll(dir, 0o750)`
で作成される。ハッシュDB は本システムの信頼の起点であり、生成時に書き込み可能な親ディレクトリや
world/group-writable なハッシュディレクトリが放置されると、攻撃者がハッシュレコードを差し替え、
後段の runner に「改ざんバイナリを検証済み」と誤認させる余地が生じる。

**要件**
record 側でも、ハッシュディレクトリおよびその祖先の権限違反を検出した場合は、無条件でフェイルクローズ
（非ゼロ終了）とし、ハッシュ生成を行わない。ハッシュディレクトリは world/group-writable を許さない
パーミッションで作成する。信頼の起点であるハッシュDB 生成では、権限違反を意図的に無視して続行する
バイパス手段を設けない。既存 `--force` は「既存ハッシュファイルの上書き」専用であり、権限違反バイパスを
兼ねさせない。

**Acceptance Criteria**:
- **AC-16**: record 実行時、ハッシュディレクトリまたはその祖先に権限違反（world/group-writable な
  書き込み可能ディレクトリ等）を検出した場合、無条件で非ゼロ終了し、ハッシュレコードを生成・保存
  しない。違反ディレクトリと具体的な権限問題（是正方法を含む）を ERROR ログに出力する。
- **AC-17**: ハッシュディレクトリが world/group-writable を許さないパーミッション
  （`0o700` 相当、または所有者 root + group 読み取り専用）で作成される。
- **AC-18**: 権限違反を意図的に無視して続行するバイパス手段（フラグ等）を設けない。既存 `--force` は
  「既存ハッシュファイルの上書き」専用であり、権限違反バイパスの意味を持たないことをドキュメントに明記する。
- **AC-19**: 「record は信頼できる管理者権限・クリーンな環境で実行すること」がドキュメントに明記される。

### F-006: ファイルサイズ上限の前提を明文化する（レポート F-6 / Info）

**現状**
`SafeReadFile` / 各アナライザは 128MB（`MaxFileSize`）を超えるファイルを拒否する。セキュリティ上は
メモリ枯渇対策として妥当だが、大型インタプリタ・巨大バイナリを検証対象にできない可用性上の制約になる。

**要件**
128MB 上限とその根拠（メモリ枯渇対策）をドキュメントに明記する。あわせて、閾値の設定可能化、または
ハッシュ計算（ストリーミング済み）と解析（サイズ制限）で上限を分離する案の可否を検討し、結論を
`02_architecture.md` に記録する。

**Acceptance Criteria**:
- **AC-20**: 128MB のファイルサイズ上限とその根拠、および可用性上の制約がユーザー向けドキュメントに明記される。
- **AC-21**: 閾値の設定可能化／ハッシュ・解析の上限分離の可否検討結果が設計文書に記録される
  （実装の要否を含めて判断する）。

### F-007: non-Linux 環境の TOCTOU 残余ウィンドウを文書で明確化する（レポート F-7 / Info）

**現状**
`openat2` 非対応環境では `safeOpenFileFallback` が「親ディレクトリの非シンボリックリンク確認 →
`O_NOFOLLOW` open → 再確認」の二段階チェックで代替する。実装は堅牢だが、原理的に openat2 の原子性には
及ばず、極めて短い競合ウィンドウが残る（コード内コメントでも認識済み）。

**要件**
本番ターゲットが Linux + openat2 前提であること、macOS 等は開発・限定用途に限る旨をドキュメントで明確化する。

**Acceptance Criteria**:
- **AC-22**: 本番ターゲットは Linux + openat2 を前提とし、non-Linux（openat2 非対応）環境には TOCTOU 残余
  ウィンドウが存在すること、および macOS 等を開発・限定用途に限る運用ガイドがドキュメントに明記される。

## 3. 非機能要件

### NF-001: 後方互換性

- native root 実行・`run_as` 未使用など、権限昇格を伴わない既存経路の挙動を回帰させない（AC-04 / AC-07 / AC-15）。
- 既存の設定ファイル・ハッシュDB 形式との互換を維持する。

### NF-002: フェイルクローズ原則

- 権限設定（補助グループ・uid/gid）、信頼の起点（record）で不確実・失敗が生じた場合は、
  安全側（実行拒否・非ゼロ終了）に倒す。fail-open を新たに導入しない。

### NF-003: テスト可能性

- 各 AC に対し、少なくとも 1 つの `test` または `static` 検証を `03_implementation_plan.md` の
  トレーサビリティ節で対応付ける。権限系（F-001/F-002）は環境依存が強いため、注入可能な syscall
  インタフェース（既存の `syscallSeteuid`/`syscallSetegid`/`identityVerifier` と同様の方式）による
  ユニットテストと、必要に応じた統合テスト方針を設計文書で定める。
- ドキュメント専用の AC（AC-12 の一部・AC-19・AC-20・AC-22）は `static` 検証（該当記述の存在確認）で足りる。

## 4. 推奨対応順序（レポート §4 準拠）

1. **F-001（補助グループ破棄）** — run-as を setuid-root で使う場合の実害が大きく最優先。
   `SysProcAttr.Credential` 移行で F-001・F-002 を同時に解決するのが望ましい。
2. **F-003（値ベースマスキング強化）** — 外部（Slack）流出面を持つため次点。
3. **F-005 / F-004** — 信頼の起点（record）と dry-run の安全側デフォルトを固める。
4. **F-006 / F-007** — 運用ドキュメントでの前提明確化と設定可能化。

> 注記: 本要件はレポート（静的レビュー）に基づく。F-001/F-002 の実害範囲は配備形態に依存するため、
> 実装着手前に対象環境での挙動を PoC で確認することを推奨する。
