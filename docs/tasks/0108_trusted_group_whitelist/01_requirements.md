# 要件定義書: ディレクトリセキュリティ検証における信頼済みグループホワイトリスト

## 1. 概要

### 1.1 背景

`record` コマンドはディレクトリのパーミションを検証し、意図せぬユーザーが書き込めるディレクトリをエラーとして扱う。現行の実装では `validateGroupWritePermissions` が「グループ書き込み可能なディレクトリ」を安全と見なす条件を `stat.Uid == 0 && stat.Gid == 0`（root:root 所有）に限定している。

macOS では `/Applications` のような標準システムディレクトリが `root:admin`（uid=0, gid=80）で `0775` のパーミションを持つ。このディレクトリは `others` に書き込みビットがない（`r-x`）ため、一般ユーザー（第三者）は書き込めない。しかし現行の検証ロジックは `Gid != 0` であることで `isRootOwned` 判定が false となり、誤ってエラーを報告する（false positive）。

`admin` グループは macOS の標準的な特権グループであり、メンバーはすでに管理者権限（`sudo` 等）を持つ。このグループへの書き込み許可は意図的な macOS の設計であり、セキュリティリスクではない。

一方で「全ユーザーが所属するグループ」（macOS の `everyone`、Linux の `users` 等）や特定多数が所属するグループへのグループ書き込みは、実質的に `others` への書き込みと同等のリスクを持つ。グループ書き込みを一律に許容するのは不適切である。

### 1.2 目的

プラットフォームごとに「信頼済み特権グループ」のホワイトリストを定義し、ホワイトリストに掲載されたグループが所有するグループ書き込み可能ディレクトリを、セキュリティ検証の例外として扱う。

### 1.3 スコープ

#### 対象

- `internal/runner/security/file_validation.go` の `validateGroupWritePermissions`
  - `isRootOwned` の条件を拡張し、「root 所有かつ信頼済みグループ」を安全と判定する
- 信頼済みグループの定義（プラットフォームごとの定数またはビルドタグ分岐）
- 上記に伴う既存テストの更新

#### 対象外

- `CanUserSafelyWriteFile`（`groupmembership` パッケージ）の変更
- `others` への書き込みビット検証ロジック（現行のまま）
- グループのメンバー数による動的判定

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| 信頼済みグループ | システム管理者のみが所属する特権グループ。メンバーはすでに別の手段でシステム全体を変更できる。macOS では `admin`（gid=80）、Linux では GID=0（`root` グループ）がデフォルト。Linux の管理者グループ（`wheel` 等）は GID がディストリビューションによって異なるため設定ファイルで指定する |
| false positive | 実際にはセキュリティリスクのないディレクトリをエラーとして報告すること |
| `isRootOwned` | 現行の「root:root 所有ディレクトリ」判定。本タスクで「root 所有かつ信頼済みグループ所有ディレクトリ」へ拡張する |

## 3. 機能要件

### FR-1: 信頼済みグループの定義

プラットフォームごとに信頼済みグループの GID セットをデフォルト値として定義する。

| プラットフォーム | グループ名 | GID | 備考 |
|-----------------|-----------|-----|------|
| macOS (darwin) | `admin` | 80 | macOS 全バージョンで固定 |
| Linux | `root` | 0 | GID=0 のみ（後方互換のデフォルト） |

> **Linux における GID の非移植性について**
>
> Linux では管理者グループ（`wheel` 等）の GID はディストリビューションによって異なる。
> GID 10 は RHEL 系・Arch Linux では `wheel` だが、Debian/Ubuntu では `uucp`（シリアル通信グループ）である。
> このため GID 10 をハードコードすると、Debian/Ubuntu 環境で `uucp` グループが root 所有ディレクトリへの書き込みを許容されるセキュリティリスクが生じる。
>
> Linux のデフォルトホワイトリストは GID=0（`root` グループ）のみとし、
> ディストリビューション固有の管理者グループを追加する必要がある場合は設定ファイルで指定する（FR-1b 参照）。

### FR-1b: Linux 向け追加信頼済みグループの設定ファイル対応

Linux 環境では、ディストリビューション固有の管理者グループ GID を設定ファイル（TOML）の `trusted_gids` フィールドで追加できるようにする。

```toml
[security]
trusted_gids = [10]  # 例: RHEL/Arch の wheel グループ
```

- 設定ファイルに `trusted_gids` が指定された場合、デフォルトホワイトリスト（GID=0）と合わせて信頼済みとして扱う。
- macOS では `trusted_gids` を無視する（`admin` GID 80 はビルドタグで固定）。
- 設定ファイルが省略された場合はデフォルトホワイトリストのみを使用する（後方互換）。

定義はビルドタグ（`//go:build darwin` 等）を用いてファイル分割し、ロジックは共通のまま保つ。

### FR-2: `validateGroupWritePermissions` の条件拡張

現行の安全条件（`stat.Uid == UIDRoot && stat.Gid == GIDRoot`）を以下のように拡張する。

**現行:**
```
安全 ← root:root 所有（uid=0, gid=0）
```

**変更後:**
```
安全 ← root 所有（uid=0）かつグループが信頼済みグループ
```

具体的には、`isRootOwned` に代わる以下の判定を実装する。

```go
// 変更後（概念）
isTrustedOwnership := stat.Uid == UIDRoot && isTrustedGroup(stat.Gid)
if isTrustedOwnership {
    return nil
}
```

`isTrustedGroup` はプラットフォーム固有の定義を参照する。GID=0（root グループ）は `isTrustedGroup` の判定に含める（後方互換）。

### FR-3: `others` 書き込みビットは従来通り拒否

本タスクはグループ書き込みに対する判定のみを緩和する。`others` への書き込みビット（`perm & 0o002`）が立っているディレクトリは引き続きエラーとする。この動作は現行実装で `validateGroupWritePermissions` の呼び出し元が担保しており、変更しない。

## 4. 非機能要件

### NFR-1: プラットフォーム非依存のコア

ホワイトリストの定義はビルドタグで分岐するが、`validateGroupWritePermissions` のロジック本体はプラットフォーム固有コードを含まない。新しいプラットフォームへの対応はホワイトリスト定義ファイルを追加するだけで完結する。

### NFR-2: 既存の厳格な動作の維持

`others` 書き込み可能ディレクトリ、および信頼済みグループ以外のグループ書き込み可能ディレクトリに対するエラー報告は変更しない。

## 5. 受け入れ基準

### AC-1: `/Applications` でエラーが発生しない（macOS）

- macOS 環境において、`/Applications`（uid=0, gid=80, perm=0775）を含むパスで `record` を実行したとき、ディレクトリセキュリティ検証エラーが発生しないこと

### AC-2: `others` 書き込み可能ディレクトリは引き続き拒否

- `perm=0777` のディレクトリを含むパスで検証を実行したとき、エラーが返ること

### AC-3: 非特権グループのグループ書き込みは引き続き拒否

- `uid=0, gid=<非信頼済みGID>, perm=0775` のディレクトリを含むパスで検証を実行したとき、エラーが返ること

### AC-4: root:root 所有ディレクトリは引き続き許容

- `uid=0, gid=0, perm=0775` のディレクトリを含むパスで検証を実行したとき、エラーが発生しないこと（後方互換）

### AC-5: Linux で `trusted_gids` を設定すると追加 GID が信頼済みになる（Linux）

- Linux 環境において、設定ファイルに `trusted_gids = [10]` を指定したとき、`uid=0, gid=10, perm=0775` のディレクトリがエラーなく通過すること
- 設定ファイルに `trusted_gids` を指定しない場合、`uid=0, gid=10, perm=0775` のディレクトリはエラーとなること（GID 10 がデフォルトホワイトリストに含まれないことの確認）

### AC-6: 既存テストの通過

- `make test` / `make lint` がエラーなしで通過すること
