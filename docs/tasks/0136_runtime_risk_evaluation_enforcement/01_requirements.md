## 1. 背景と目的

### 背景

PR #724（コマンドリスク判定の技術ドキュメント追加）のレビューを契機に、ドキュメントと実装の差異を徹底的に調査した結果、`runner` の **実行時リスク判定（`risk.StandardEvaluator.EvaluateRisk`）** とその周辺に複数の不具合・仕様不整合が確認された。これらの多くは「実行時に算出される実効リスクが、設計意図より低くなる」「ドキュメントが存在しない保証を約束する」方向の問題を含み、セキュリティ上の影響がある。

確認された事象（A〜F は PR #724 のレビューコメント、G〜J は本調査での追加発見）:

### 目的

1. 実行時リスク判定が、設計意図どおりに **安全側（fail-safe / fail-closed）** で動作することを保証する（A, D, E, G）。バイナリの素性を確認できない不確実ケースは、設定によらず実行を中止する。
2. コマンドごとのリスク定義の **一貫性（単一の真実の源泉）** を確立し、実行時・dry-run・ドキュメントの分裂をなくす（H）。
3. リスク判定結果の **監査可能性** を確保する（B）。
4. dry-runが実行時の判定（実効リスクと実行可否）を正しく予告できるようにする（I）。
5. 開発者向け・ユーザー向けドキュメントおよびコードコメントを実装の実態と一致させる（C, E, F, J）。

### スコープ外

- グループレベル / グローバルレベルの `risk_level` 継承の **新規追加**（C はあくまで現状スコープの正確な記述に留める）。
- リスクレベルの段階定義（Unknown/Low/Medium/High/Critical）の変更。
- バイナリ静的解析アルゴリズム自体の変更。
- root 実行コマンド向けの危険判定（`IsDangerousRootCommand` 等）の変更。

- AC-15 / AC-17 / AC-18 が改訂対象とする `docs/dev/architecture_design/command-risk-evaluation.ja.md` / `.md` は **PR #724（未マージ）にのみ存在** する。本タスクのドキュメント改訂作業は PR #724 のマージ後に行うか、マージ順序を実装計画で調整する。

### 脅威モデルと判定方式の前提（セキュリティレビュー反映）

リスク判定は **多層防御の 1 層** であり、単独で全脅威を遮断するものではない。本タスクの判定強化は以下の前提・限界の上に成立する。これらは設計文書・ユーザー文書に明記し（AC-66 / AC-67）、利用者が誤った安心を持たないようにする。

1. **ブロックリスト方式である**: リスク判定は「既知の危険シグナル」を加点する方式であり、プロファイル・パターン・解析のいずれにも該当しない **未知のコマンドは Low と算出され、既定 `risk_level = "low"` で通過し得る**（F-005 が救うのはバイナリ解析の不確実のみ）。安全な運用には **allowlist（実行可能コマンドの明示制限）と verify_files によるハッシュ固定の併用が前提** である。本前提を要件として明記する。
2. **判定は基本的に argv[0]（コマンド名）中心**: F-013 で間接実行（ラッパー／シェル）を安全側に倒すが、リスクプロファイルの照合は basename ベースであり、**ハードリンク・リネームによる名前すり替えには対応しない**（シンボリックリンクは F-012 で解決）。検証済みバイナリでも「危険バイナリを別名で配置」したケースのリスクは正しく出ない。緩和は allowlist + ハッシュ固定。
3. **判定対象外の副作用がある**: `output_file` による任意パスへの書き込みは、書き込み先のリスク寄与を実行時リスク判定が評価しない（出力パス検査は traversal 検査のみ）。本タスクのスコープ外だが、限界として明記する。
4. **root 実行向けの危険判定は別系統**: 部分一致方式で、通常の実行時リスク判定（完全一致方式）と方針が異なる。本タスクでは統合しないが、両系統の適用パスと優先順位を文書化する（F-008 / AC-67）。

**Acceptance Criteria（脅威モデル明記）**:
- **AC-66**: リスク判定がブロックリスト方式であり、未知コマンドは Low 通過し得ること、安全運用には allowlist + ハッシュ固定の併用が前提であることが、開発者向け・ユーザー向けドキュメントに明記される。
- **AC-67**: basename 方式の限界（ハードリンク/リネーム非対応）、`output_file` 書き込み先の判定対象外、root 判定系との関係（適用パス・優先順位）が、脅威モデルとして文書に明記される。

## 2. 機能要件

### F-001: 実行時リスク判定がコマンドリスクプロファイルの全要因を反映する（A）

実行時リスク判定は、コマンドに対応するリスクプロファイルが存在する場合、その **全リスク要因の最大値** を実効リスクに反映しなければならない。プロファイルに宣言された任意のリスク要因（特権昇格・ネットワーク・破壊操作・データ持ち出し・システム変更）が、実行時の実効リスクに正しく寄与する。

リスクの算出は **安全側へのみ作用** する。すなわちプロファイル反映によって実効リスクが下がることはなく、既存の各ステップ判定結果とプロファイルの `BaseRiskLevel()` の **最大値** を採用する。

不変条件として、コマンドが複数のリスク次元（特権昇格・破壊操作・ネットワーク・システム変更・データ持ち出し・引数による危険・バイナリ解析由来の危険等）に該当する場合、**該当する次元のうち最も高いリスクを実効リスクとする**。すなわち実効リスクは評価の順序に依存せず、ある判定が先に確定して別のより高い危険次元を取りこぼすことがない。

なお、引数によってリスクが変わる次元（F-011 のシステム変更等）については、**引数評価後のリスク**が上記の最大値に反映される。詳細は F-011 を参照。

**例外（coreutils）**: coreutils 単一バイナリ配下のコマンドは coreutils 分類が権威を持ち、**バイナリ解析の次元を最大値の対象から除外する**（共有バイナリのシンボル/syscall は個々のサブコマンドの挙動を表さないため）。詳細・優先順位は F-008 を参照。

**Acceptance Criteria**:
- **AC-01**: `claude`（`DataExfilRisk=High` / `NetworkRisk=High` を宣言）の実行時実効リスクが **High** になる（従来は Medium）。
- **AC-02**: プロファイルで宣言された各リスク要因の最大値が実効リスクの下限となる（例: 任意の要因が High を宣言するコマンドは実効リスクが High 以上）。
- **AC-03**: プロファイル反映は安全側へのみ作用し、プロファイルが存在しても他ステップ（特権昇格→Critical 等）の結果より低い値に上書きされない。
- **AC-04**: **F-001 のプロファイル反映ステップ単体では**、プロファイルが存在しないコマンドの実効リスクを変化させない（F-002/F-005/F-007 等の他要件による変化はこの AC の対象外）。
- **AC-05**: コマンド名はシンボリックリンクを解決して照合する。リンク連鎖上の複数の名前が異なるプロファイルに一致した場合は、一致した全プロファイルのリスクの **最大値** を採用する。ただしリンク解決に失敗（リンク先取得失敗・循環・深度超過・解決不能）が混在する場合は、部分的に解決した名前の最大値で続行せず F-012 の安全側中止を **優先** する。
- **AC-22**: `systemctl` / `service`（`SystemModRisk=High` を宣言）の **変更系サブコマンド**（F-011 参照）の実行時実効リスクが **High** になる（従来は実行時（絶対パス解決後）Low、basename 入力時 Medium）。この振る舞い変更は意図的なものとして移行ノート（AC-19）に記載される。読み取り専用サブコマンドの扱いは F-011 / AC-49 を参照。
- **AC-63**: 複数のリスク次元に該当するコマンドの実効リスクが、各次元単独のリスクの **最大値** になる（評価順に依存しない）ことをテストで検証する。coreutils 分類適用時のバイナリ解析次元の除外は F-008 / AC-72 に従う。

### F-002: 名前ベースの判定が解決済み絶対パスに対しても機能する（D）

破壊的操作・システム変更の判定は、実行時に渡される **解決済み絶対パス**（例: `/usr/bin/rm`）に対しても、対象コマンドを正しく検出しなければならない。判定はコマンド名（basename）に対して行い、シンボリックリンクの解決を考慮する。

**Acceptance Criteria**:
- **AC-06**: 絶対パスで与えられた破壊的コマンド（例 `/usr/bin/rm`）が破壊的操作として分類される（basename だけでなく絶対パスでも検出される）。
- **AC-07**: 絶対パスで与えられたシステム変更コマンド（例 `/usr/sbin/systemctl restart …`）がシステム変更として分類される。
- **AC-08**: 絶対パスで指定された `/usr/bin/rm -rf /tmp/x` の実効リスクが **High** になる（従来は Low）。
- **AC-09**: 部分一致による誤検出が起きない（例: `/usr/bin/lsrm` や `/usr/bin/systemctl-helper` は `rm` / `systemctl` と一致しない）。
- **AC-10**: F-002 によるパス対応の追加が、既存の basename 入力に対する **検出能力** を損なわない（例: basename `rm` が引き続き破壊的操作として検出される）。本 AC は「検出の後方互換」であり、本タスクで意図的に引き上げる対象（`systemctl` 変更系→High、インタプリタ・ビルドツール→High 等）のリスクレベルを据え置くことを意味しない。
- **AC-23**: リスク判定のテストは、実行時の実態（コマンドが解決済み絶対パスで渡されること）を反映した絶対パス入力のケースを含む。
- **AC-44**: プロファイルを持たないコマンド（例: `rmdir`, `shred`, `mount`, `crontab`）の絶対パス入力について、正しいリスク（破壊的→High、システム変更→Medium）が算出されることをテストで検証する（F-001 のプロファイル反映では救えない、F-002 固有の効果の分離検証）。
- **AC-62**: `find` の `-exec` 対象コマンドおよび `rsync --delete` 等の判定が、**絶対パス・coreutils 配下・basename** のいずれの形でも機能する。例: `find . -exec /usr/bin/rm {} \;` および coreutils 配下 rm を exec 対象とするケースを破壊的と判定する（`-exec` 対象にも basename 正規化・シンボリックリンク解決を適用する）。

### F-012: シンボリックリンク解決の信頼境界を安全側に固める（A/D 修正の随伴）

リスク判定はコマンド名の照合のためシンボリックリンク連鎖を解決する（F-001 AC-05 / F-002）。リンク解決に失敗した際に、解決できた部分的な名前だけで評価を続行すると、危険な実体名を観測できないまま Low 側に落ちる抜け穴になり得る（現状はこの形で黙って続行する。詳細な現状挙動は背景・02 を参照）。

シンボリックリンク解決に関わる以下の異常時は、部分結果で続行せず **安全側（実行中止: エラー中止または内部 Critical 化）** としなければならない。「解決できた名前だけで評価して続行」することを禁止する。

- リンク先の取得失敗（権限不足・TOCTOU でのファイル消失等）
- シンボリックリンクの循環
- 解決深度の上限超過
- 最終ターゲットの解決不能

なお、実行時パスでは正常系のパス解決が先行するため、本要件の対象は「リスク判定内のリンク走査」における異常検知と安全側動作とする。正常系で追加のリンクが無いケース（解決済み絶対パスが通常ファイル）は従来どおり正常に評価する。

**Acceptance Criteria**:
- **AC-54**: シンボリックリンク解決の失敗（リンク先取得失敗・循環・深度超過・解決不能）時、リスク判定は部分的に解決した名前だけで評価を続行せず、安全側（エラー中止または内部 Critical 化）とする。
- **AC-55**: 上記の解決失敗時に実効リスクが Low 側に落ちないことをテストで検証する（深度超過だけでなくリンク先取得失敗のケースを含む）。

### F-003: リスク判定結果が監査ログに記録される（B）

実行時にコマンドのリスク判定を行った際、その結果が監査ログ（`command_risk_profile` エントリ）として出力されなければならない。インシデント調査に耐えるよう、「どのバイナリ・どの設定・どの解析レコードで、許可/拒否のいずれを下したか」を後から相関できる情報を含める。

**実現上の注記**: 本要件は、リスク判定の根拠（機械可読な reason code）を判定処理から監査層へ伝える仕組みと、監査ロガーの配線を要する。`decision`（allow/deny）や `max_allowed_risk` はリスク比較を行う層でのみ記録できる。具体的な実現方式（評価器インターフェースの拡張・配線箇所等）は 02_architecture で確定する。

**Acceptance Criteria**:
- **AC-11**: 通常実行パスでコマンドのリスク判定が行われると、`audit_type=command_risk_profile` の監査ログエントリが 1 件出力される。
- **AC-12**: 出力エントリには、評価されたリスクレベルと判定根拠が含まれる。判定根拠は **機械可読な reason code**（例: `destructive_file_operation` / `binary_analysis_dynamic_load` / `coreutils_classification` / `profile_data_exfil` / `uncertain_missing_record`）として出力され、プロファイル由来のコマンドでは加えて人間可読なリスク要因の理由（`risk_factors`）を含む。
- **AC-13**: ログレベルがリスクレベルに対応する（Critical→Error, High→Warn, Medium→Info, それ以外→Debug）。
- **AC-48**: プロファイルを持たないコマンド（例: バイナリ解析で Medium となった未知コマンド）についても、AC-12 の判定根拠（reason code）が出力される。
- **AC-69**: 異なる判定根拠（ネットワーク検出・動的ロード・exec・直接 syscall・mprotect・解析非対応・解析エラー・不確実など）には **それぞれ異なる reason code** が割り当てられ、「なぜそのリスクか」が根拠ごとに区別できる。全種別が網羅的にテストされる。
- **AC-14**: リスク超過で実行拒否されたコマンドについても、リスク判定結果が監査可能である（拒否ログまたはリスクプロファイルログのいずれかで追跡できる）。
- **AC-56**: 監査ログのリスク判定エントリは、相関に必要な以下のフィールドを含む: `resolved_path`、`content_hash`、解析レコードの識別情報（レコードの hash または schema version。レコード非使用時はその旨を示す明示値）、`max_allowed_risk`、`decision`（allow/deny）、`reason_codes`。値が取得できる場合は **空でない実値** を記録する。ただしパス解決失敗・ハッシュ検証前の拒否などで `resolved_path` / `content_hash` 等が取得できない場合は、**明示的なセンチネル値**（例: `unresolved` / `unverified`）を記録する（バリデーションエラーにはしない）。リスク超過・不確実・特権昇格・解決失敗などで **拒否（deny）された場合にも必ず出力される**（早期 return で監査が抜けない）。
- **AC-57**: コマンド引数を監査ログに出力する場合、機密情報の漏えいを防ぐマスキング方針（既存の redaction 機構との整合を含む）が定義され、出力に適用される。
- **AC-70**: 全ての拒否（deny）系イベント（リスク超過・不確実ケース・特権昇格ブロック）が、一貫したログレベル・`audit_type` で相関検索できる（インシデント調査時に拒否事象を一括抽出できる）。

### F-004: `risk_level` の設定スコープを実態どおりに記述する（C）

`risk_level` の設定可能スコープ（コマンドレベル＋テンプレートフォールバック、グループ/グローバル非対応、未指定時は `low`）について、ドキュメントおよびコードコメントを実態と一致させる。存在しない「グローバルデフォルトからの継承」を示唆する記述を排除する。

**Acceptance Criteria**:
- **AC-15**: コマンドリスク判定ドキュメント（`command-risk-evaluation.ja.md` / `.md`）が、`risk_level` はコマンドレベル（テンプレートフォールバックあり）でのみ設定可能で、グループ/グローバルレベルでは設定できないことを明記する。
- **AC-16**: 誤解を招くコードコメント（`spec.go` の `CommandTemplate.RiskLevel` の `nil: inherit from global default`、`CommandSpec.RiskLevel` の `nil=inherit default`）が、実態（`nil` の場合はテンプレートフォールバックの後、最終的に `low`）に即した記述へ修正される。

### F-005: バイナリ解析の不確実ケースでは実行を中止する（E）

バイナリ解析の **不確実ケース**（バイナリの素性・解析結果の信頼性を確認できない状態）では、`risk_level` の設定値によらずコマンドの実行を中止しなければならない。実現機構としては **内部的に Critical 扱いに昇格** させる方式を第一候補とする（Critical は設定不可のため、統一されたリスク比較パス `effectiveRisk > maxAllowedRisk` で必ず拒否され、拒否ログ・監査の対象にもなる）。エラー返却方式との比較・選定はアーキテクチャ設計で行う。

**実現上の注記**: 現状は「危険検出（High 維持の対象）」と「不確実（中止の対象）」を区別せず同一に扱っているため、両者を区別して伝える必要がある。具体的な伝搬方式は 02_architecture で確定する。

**不確実ケース（実行中止の対象）** — 前提条件: バイナリ解析が有効であること:
- バイナリの同一性が未検証（防御的分類。本番配線では通常発生しない。背景 E 参照）
- 解析レコードが見つからない（ハッシュ検証済みバイナリにレコードがない。静的バイナリで syscall 解析レコードが無い場合を含む）
- 解析レコードのスキーマバージョン不一致
- 解析レコードとディスク上のバイナリの不一致（解析 DB の陳腐化）
- 解析非対応の形式・解析処理の失敗（記録時と実行時のバイナリ不一致を含む）・想定外の解析結果

なお「静的バイナリ」は動的シンボルが無いだけで syscall 解析により判定可能であり、本質的な不確実ではない。不確実となるのは上記のとおり **解析データ自体を入手できない場合** に限る（区分の網羅は AC-45 を参照）。

**coreutils 分類との関係**: coreutils 単一バイナリ配下のコマンドは coreutils 分類がバイナリ解析より優先する（F-008）。したがって、本要件の不確実ケース（解析レコード欠落・スキーマ/ハッシュ不一致等の **バイナリ解析次元**）は **coreutils 分類が適用されるコマンドには適用しない**（安全サブコマンド `echo` は解析レコードが無くても Low）。ただしバイナリ自体の **ハッシュ未検証** は F-014 により引き続き不確実として扱う（バイナリ解析次元の抑制とハッシュ検証は別）。

**解析無効時（バイナリ解析が無効な構成）の扱い**:
- セキュリティ境界として見ると、解析無効は「危険シグナルを観測できない状態」であり、黙ったフェイルオープン（Low 側通過）は不適切である。
- バイナリ解析が無効な構成では、**コマンドの実行を常に拒否する**。これを回避して実行を許可するオプトイン機能（`allow_unanalyzed_binaries` 相当）は **設けない**（`risk_level` の値によらず実行できない）。
- **dry-run は可能** とする（実行しないため）。dry-run では解析無効である旨と、実行時には拒否される旨を表示する。
- 解析無効による拒否はログ・監査で明示する。

**High のまま維持するケース（`risk_level = "high"` で意識的に許可可能）**:
- 有効な解析レコードから検出された危険な性質: 動的ロードシンボル（`dlopen`/`dlsym`/`dlvsym`）、exec syscall、未解決の `svc #0x80` 直接 syscall、mprotect 系 `PROT_EXEC`（確定または不明）
- これらは「解析が成功し、バイナリの性質として確認・推定された」ものであり、利用者がリスクを受容して許可する運用（ログメッセージが案内する `set risk_level = "high" or higher to allow execution`）を維持する

**Acceptance Criteria**:
- **AC-40**: 上記の不確実ケース各々について（前提条件下で）、`risk_level = "high"` を設定していてもコマンドが実行されない。採用する実現機構（内部 Critical 化 / error 返却）に依らず、不確実ケースは必ず実行されない。
- **AC-41**: 不確実ケースによる実行中止は、拒否理由（どの不確実ケースか）とともにログ・監査で追跡できる。
- **AC-42**: 有効な解析レコードから検出された危険な性質（動的ロード・exec・svc・mprotect）は従来どおり High であり、`risk_level = "high"` で実行できる。
- **AC-43**: dry-runも同じ分類を表示する（不確実ケースは実行拒否予告。既存の「ハッシュ検証失敗 → Critical」表示と整合する）。normal 実行と dry-run で不確実ケースの判定が一致する（機構選定による分岐がない）。dry-run 固有の制約は AC-46 を参照。
- **AC-51**: バイナリ解析が無効な構成では、(a) コマンドの実行が **常に拒否** される、(b) これを回避して実行を許可するオプトイン機能は存在しない（`risk_level` の値によらず実行不可）、(c) dry-run は可能で、解析無効である旨と実行時に拒否される旨が表示される、(d) 解析無効による拒否がログ・監査で明示される、(e) この常時拒否は **coreutils 分類コマンド（`echo`/`ls` 等）にも適用される**。coreutils のバイナリ解析次元の抑制（F-008）は「バイナリ解析シグナルを無視する」ことであって「ハッシュ検証を免除する」ことではないため、解析（＝ファイル検証）が無効でバイナリの同一性を確認できない構成では、coreutils コマンドも実行拒否となる。
- **AC-45**: バイナリ解析が返し得る **全ての結果区分** について「不確実（中止）/ 危険検出（High）/ 安全側通過」のいずれに属するかが設計文書に明記され、テストで検証される。区分の扱いは次のとおり:
  - 危険シグナル検出（ネットワーク/動的ロード/exec/直接 syscall/mprotect 等）→ **危険検出（High）**
  - 解析が成功し危険シグナルなし → **安全側（Low）**
  - **解析データを入手できない**（解析レコード欠落、静的バイナリで syscall 解析レコードが無い、解析非対応の形式、想定外の結果）→ **不確実（中止）**。fail-open で Low に落とさない（F-005）。
  - 解析エラー・ハッシュ不一致 → 不確実（中止）/ エラー
  - **補足（静的バイナリの正しい理解）**: 「静的バイナリ」は動的シンボルが無いだけで、**syscall 解析で判定可能**（むしろ動的リンクより直接的）。`record` 時に syscall 解析データが生成されていれば静的バイナリも通常どおり分類される。したがって「静的だから一律 Low 通過」も「静的だから一律ブロック」もせず、**得られた解析データに基づいて判定し、データが無い場合のみ不確実として中止**する。
  - **前提（実装計画で確認）**: `record` が静的バイナリに対しても syscall 解析データを生成すること。これにより正当な静的バイナリは分類可能となり、データ欠落時のみ中止となる（恒久的な全静的バイナリ拒否を避ける）。
- **AC-17**: 開発者向けドキュメントが、(a) エラーで実行中止するケース（シンボリックリンク深度超過・coreutils のファイル情報取得エラー・解析処理自体の失敗ではない予期しないレコード読込エラー）、(b) 不確実ケースとして設定によらず実行中止するケース（本要件）、(c) High として許可設定下で実行可能なケース、(d) 解析無効時の挙動、を明確に区別して記述する。ユーザー向け `risk_assessment.ja.md` §3.3 の表も新しい挙動に合わせて改訂する。

### F-006: dry-runのエラー時挙動を正確に記述する（F）

dry-runパスはパス解決失敗・解析エラー時にエラーを返す（High 表示で継続しない）。一方、dry-run の詳細解析内の特定チェックは失敗を High/Critical に変換する。この区別をドキュメントに反映する。

**Acceptance Criteria**:
- **AC-18**: ドキュメントの実行時パス/dry-runパス比較表が、dry-runはパス解決・解析の失敗時にエラーを返すこと、High/Critical への変換は dry-run の詳細解析内の特定チェックに限られることを正確に記述する。

### F-007: 設定値 `"unknown"` を拒否する（G）

`ParseRiskLevel` は、設定値 `"unknown"` に対して `"critical"` と同様にエラーを返さなければならない。`risk_level` として受理される値は `"low"` / `"medium"` / `"high"`（および省略・空文字 = `low`）のみとする。

**Acceptance Criteria**:
- **AC-24**: `ParseRiskLevel("unknown")` がエラーを返す（`ErrInvalidRiskLevel` 系）。
- **AC-25**: `risk_level = "unknown"` を含む TOML 設定は、設定ロード時または実行前検証でエラーとなり、その旨が利用者に通知される。
- **AC-26**: `"low"` / `"medium"` / `"high"` / 省略 / 空文字の挙動は従来どおり維持される。

### F-008: コマンドごとのリスク定義の一貫性を確立する（H）

同一コマンドに対するリスク定義（コマンドリスクプロファイル / コマンド名ベースの破壊・システム変更判定 / 引数パターンによる危険判定 / coreutils 分類）の間で、定義の重複に起因する判定の分裂をなくす。アーキテクチャ設計で「単一の真実の源泉」または「明示的な優先順位」を定め、実行時とdry-runが同一コマンド・同一引数に対して矛盾したリスクを返さないようにする。

**実行時の引数評価は必須**: 実行可否を決めるリスク判定は、コマンド名だけでなく **引数パターンも必ず評価** しなければならない。危険な引数パターン（`rm -rf`、`dd if=`、`chmod 777` 等）は dry-run 専用の参考情報ではなく、実効リスク算出の入力に統合する。これにより「dry-run だけが危険な引数を検出し、実行時は見逃す」状態の再発を防ぐ。

**coreutils 分類の優先（重要・タスク 0135 の設計保持）**: coreutils 単一バイナリ配下のコマンドについては、**coreutils 分類がバイナリ解析より優先**する。共有バイナリのシンボル/syscall は個々のサブコマンドの挙動を表さないため、coreutils 分類が適用されるコマンドでは **バイナリ解析の次元（その High シグナルおよび解析レコードの不確実 = F-005）を抑制** する。すなわち安全サブコマンド（例 `echo`）は共有バイナリに dlopen 等があっても・解析レコードが無くても Low、破壊サブコマンド（例 `rm`）は High となる。ただしこれは **バイナリ解析次元の抑制**であって、バイナリ自体の **ハッシュ検証は引き続き必要**（未検証は F-014 により不確実扱い）。F-001 の最大値もこの抑制後の次元集合に対して計算する。

**Acceptance Criteria**:
- **AC-27**: `systemctl`（引数任意）について、実行時とdry-runが同じリスクレベルを返す。
- **AC-28**: `rm`（破壊的操作の代表）について、basename 入力・絶対パス入力・coreutils 配下のいずれでも、実行時とdry-runで High が返る。
- **AC-29**: 複数機構に定義が重複するコマンドの優先順位（どの定義がどのパスで効くか）が、設計文書および開発者向けドキュメントに明記される。root 実行向けの危険判定（部分一致方式）と通常の実行時リスク判定（完全一致方式）の **適用パスと優先順位の関係** も文書化する（統合は本タスクスコープ外だが、関係の明示は行う）。
- **AC-68**: coreutils マルチコール（`coreutils <subcmd>`）のサブコマンド判別が、破壊的サブコマンド（`rm`/`dd`/`shred` 等）を確実に検出する。サブコマンド判別不能・未知の場合は、破壊的サブコマンドを含み得るため **High** とする（明示的に allowlist された「サブコマンドなしの安全ケース」のみを例外として許容してよい）。Medium 既定で `coreutils <未解析>` が `risk_level = "medium"` 下で実行されることを防ぐ。
- **AC-47**: 引数パターンによってのみ危険と判定されるコマンド・引数の組について、実行時とdry-runの実効リスクが分裂しないよう定義が統合され、テストで検証される。少なくとも以下の具体ケースを含む: `chmod -R 777 /`（High）、`chown -R root /tmp/x`（Medium 以上）、`mkfs.ext4 /dev/sdX`（High）、`dd if=...`（High）。いずれも **実行時のリスク判定が引数を評価して** 該当リスクを返すこと（現状は dry-run のみ引数パターンが効き、実行時はより低いリスクに落ちる）。
- **AC-72**: coreutils 単一バイナリ配下の安全サブコマンド（例 `echo`）は、共有バイナリにバイナリ解析由来の High シグナル（dlopen/exec 等）があっても、また解析レコードが欠落していても、（ハッシュ検証済みである限り）実効リスクが **Low** になる。破壊サブコマンド（例 `rm`）は同条件で **High** になる。すなわち coreutils 分類がバイナリ解析より優先することをテストで検証する（タスク 0135 の設計保持）。

### F-009: dry-runが実行時の判定を予告できる（I）

dry-runは、実行時と同じロジックで算出した実効リスクと、`risk_level` 設定との比較結果（実行時に許可されるか拒否されるか）を表示しなければならない。既存の詳細表示（ディレクトリ既定リスク・ハッシュ検証・setuid・危険パターン等）は維持してよいが、実行可否を決めるのは実効リスク表示の側であることを出力上明確にする。

**dry-run 固有の制約**: dry-run はハッシュディレクトリが利用不能でも続行する設計（警告のみで検証スキップ）があり、その場合 **検証済みハッシュが得られず**、実行時と同一の判定を再現できない。このとき F-005 の不確実ケース（Critical/拒否予告）と誤って断定してはならない。

**Acceptance Criteria**:
- **AC-30**: dry-run出力に、実行時パスと同一ロジックで算出された実効リスクが含まれる（ハッシュ・解析レコードが利用可能な場合）。
- **AC-31**: dry-run出力に、`risk_level` 設定との比較結果（許可/拒否の予告）が含まれる。
- **AC-32**: バイナリ解析シグナル（ネットワーク/動的ロード/exec/mprotect）による High/Medium が、dry-runの実効リスク表示にも反映される（解析レコードが利用可能な場合）。
- **AC-33**: dry-runは引き続きコマンドを実行しない。**ハードエラー**（パス解決失敗・コマンド不在など、解析自体に進めないケース）は AC-18 のとおりエラーを返す。一方、**報告可能な状態**は preview として出力し中断しない: ハッシュ未取得 → `unknown`（AC-46 / AC-58）、解析が利用可能な状況での不確実 → 拒否予告（AC-43）。ハードエラーと報告可能状態（`unknown` / 拒否予告）を明確に区別する。
- **AC-46**: ハッシュディレクトリ利用不能などで dry-run が検証済みハッシュを取得できない場合、不確実ケース（Critical/拒否予告）と **誤断定せず**、`unknown`（判定不能: ハッシュ未検証のため実行時に再評価）として、allow/deny とは区別された分類で表示する。
- **AC-58**: dry-run の `unknown`（判定不能）は、出力上 allow/deny と明確に区別され、終了コードが通常成功と区別される（または区別するオプションがある）。dry-run 出力には「dry-run は本番実行時の許可を保証しない（本番では再評価され拒否され得る）」旨を含める。CI 利用向けに、`unknown` を失敗（非ゼロ終了）として扱うオプションを提供する。

### F-011: 読み取り専用サブコマンドの過剰な High 化を避ける（A 修正の随伴）

F-001 により `systemctl` / `service` プロファイルの `SystemModRisk=High` が実行時に反映されると、読み取り専用の典型用途（`systemctl status` / `is-active` / `list-units` 等による監視）まで一律 High となり、利用者は監視コマンドに `risk_level = "high"` を設定せざるを得なくなる。これは最小権限の原則（ユーザー向け文書 §5 が掲げる「実際の動作に必要な最低限のリスクレベルを設定する」）を利用者自身が損なう誘因（リスク補償）になる。

これを避けるため、システム変更リスクに **引数（サブコマンド）条件付き評価** を導入する（ネットワークの条件付き判定と同様の引数条件付き機構）。ただし読み取り専用を「昇格なし（Low 落ち）」にはせず **Medium の下限** を持たせる（`systemctl show` 等はユニット設定・パス・ユーザー名等の情報露出になり得るため、Low は危険）。

- 変更系サブコマンド（`start` / `stop` / `restart` / `enable` / `disable` / `mask` 等）→ High
- 読み取り専用サブコマンド（`status` / `show` / `cat` / `is-active` / `list-units` / `list-unit-files` / `list-timers` / `list-dependencies` 等）→ **Medium 下限**（プロファイルが Medium を保証し、Low には落とさない。他ステップがより高いリスクを返す場合はその最大値）
- **未知のサブコマンド・サブコマンド判別不能 → High（安全側既定）**

**F-001（最大値）との関係（重要）**: F-001 の「最大値」に用いるシステム変更リスクは、**引数評価後の値**（`status`→Medium、`restart`→High 等）でなければならない。引数を見ない一律 High を最大値に持ち込むと `systemctl status` まで High に固定され、F-011 の目的（最小権限）と AC-49 が破綻する。引数条件評価を最大値計算に正しく織り込む実現方式は 02_architecture で確定する。

> 情報露出量の多い読み取り系（`show` / `cat` / `list-unit-files` / `list-timers` / `list-dependencies` 等）を Medium とすることを明記する。読み取り専用/変更系の **網羅的な分類リスト** とその管理方針（新規サブコマンド追加時の既定 = High）はアーキテクチャ設計で確定する。

> ✅ **決定事項（2026-06-14）**: 条件付き評価の導入を採用する（無条件 High 案は不採用）。代替案として「無条件 High を受け入れ、ユーザー向け文書の設定例を `risk_level = "high"` に改訂する」案も検討したが、最小権限の原則を損なわないため条件付き評価を選択した。読み取り専用は Low ではなく Medium 下限とする（情報露出対策）。

**Acceptance Criteria**:
- **AC-49**: `systemctl status <unit>` / `systemctl show <unit>` の実行時実効リスクが High にならず、かつ **Low にも落ちず Medium 以上** になる。`systemctl restart <unit>` および未知サブコマンドは High になる。
- **AC-75**: `service` についても引数条件付き評価を行う。ただし引数構造が異なる（`service <name> <action>` で **アクションは unit 名の後**）ため、アクション引数で判定する: `service nginx status`→Medium 下限、`service nginx restart`→High、未知アクション・アクション判別不能→High。systemctl と service の引数位置の差を設計で吸収する。

### F-013: 間接実行（ラッパー／シェル）をリスク判定でバイパスさせない（セキュリティレビュー C-1/C-2/H-3）

リスク判定は argv[0]（コマンド名）と引数を見るため、実体が危険なコマンドを無害なラッパー／シェルでくるむと、リスク判定（および特権昇格の Critical ゲート）を回避できる。

- ラッパー例: `cmd=/usr/bin/env, args=[sudo, rm, -rf, /]` → argv[0]=`env` が特権昇格・破壊いずれにも非該当で Critical/High を回避。`timeout 60 systemctl stop nginx`、`xargs rm`、`nice dd ...` も同様。
- シェル例: `cmd=/bin/bash, args=[-c, "rm -rf / ; systemctl stop firewalld"]` → Medium 止まり（`-c` の中身は 1 個の文字列引数で危険パターン非該当）。

中核ゲート（特権昇格の遮断・破壊操作の High 化）の自明な迂回を塞ぐため、間接実行を安全側に評価する。フル汎用のラッパー解析は重量級のため、本要件では **フェイルセーフな最小要件** を定める（網羅的なラッパー一覧と再帰的アンラップの範囲はアーキテクチャ設計で確定）。

- **ラッパーコマンド**（`env` / `nice` / `ionice` / `timeout` / `nohup` / `stdbuf` / `setsid` / `xargs` / `time` / `chrt` / `taskset` 等）が argv[0] の場合: (a) 実引数から実コマンドを抽出して再帰的にリスク評価する、または (b) 抽出が不能・未対応なら安全側（最低 High）に倒す。
- **抽出した実コマンドも本来のゲートを通す**: (a) で抽出したインナーコマンド（例: `env rm` の `rm`）は、通常のコマンドと同様に **allowlist（`cmd_allowed`）照合とハッシュ検証** の対象とする。これらを通せない場合は実行拒否とする（インナーコマンドが未許可・未検証のまま実行され、allowlist/検証をすり抜けることを防ぐ）。
- **ラッパーを実コマンドなしで起動した場合**（例: `env` 単体・`nice` 単体・`timeout --help`）は「抽出不能（実コマンド指定があるが抽出に失敗）」とは区別する。ただし一律 Low にはしない: `env` 単体は環境変数（機密を含み得る）を出力するため安全側（最低 Medium）とする等、コマンド固有に扱う。区分の詳細はアーキテクチャ設計で確定し、判別に確証がなければ安全側に倒す。
- **特権昇格の独立ルール（抽出可否に依らず適用）**: 間接実行の引数列に `sudo` / `su` / `doas` が **独立したコマンドトークンとして** 出現する場合は、実コマンド抽出の成否に依らず **Critical** とする（例: `env sudo …` / `xargs sudo`）。これらが抽出失敗時の High 下限へ落ちて Critical を回避することを防ぐ（AC-59 の保証根拠）。
- **シェル／インタプリタのインラインコード実行**（`sh -c` / `bash -c` / `python -c` / `perl -e` 等）: インライン文字列は静的に安全解析できないため、実効リスクの **High 下限** とする（Medium 止まりにしない、F-015）。**限界（明記）**: インライン文字列の中身（例: `bash -c "sudo ..."`）は不透明で確実に解析できないため、文字列内に隠れた特権昇格を Critical に確実に引き上げることは保証しない（best-effort の検出は妨げない）。これは脅威モデルの限界として文書化する（allowlist 併用が前提）。
- `env` を介した変数設定後のコマンド（`env FOO=bar <cmd> …`）など、実コマンドの抽出規則の詳細はアーキテクチャ設計で定める。抽出規則が曖昧なケースは安全側（High 以上）とする。

**Acceptance Criteria**:
- **AC-59**: `env sudo …` / `timeout 60 sudo …` / `xargs sudo` の実効リスクが **Critical**（または実行中止）になり、特権昇格の遮断がラッパーで回避できない。
- **AC-60**: ラッパー経由の破壊的・システム変更コマンド（例: `env rm -rf /tmp/x`、`timeout 60 systemctl stop nginx`、`nice dd if=…`）の実効リスクが、ラップしない場合と同等（破壊→High、システム変更→該当レベル）以上になる。未対応・抽出不能なラッパー実行は安全側（High 以上）に倒れる。
- **AC-61**: シェル／インタプリタのインラインコード実行（`bash -c "…"` / `python -c "…"` 等）の実効リスクが Medium 止まりにならず **High 以上** になる（F-015 によりインタプリタ・シェルは一律 High。本 AC はインラインコードの特例の明示）。
- **AC-71**: 上記バイパス系（AC-59〜AC-61、AC-62）について、攻撃者視点の回避ケースとしてテストが追加される（正常系の直接呼び出しだけで終わらせない）。
- **AC-77**: ラッパーから抽出した実コマンド（例: `env rm -rf …` の `rm`）が、通常のコマンドと同じ allowlist（`cmd_allowed`）照合・ハッシュ検証の対象となり、未許可・未検証のまま実行されない。これらを通せない場合は実行拒否となることをテストで検証する。
- **AC-78**: ラッパーを実コマンドなしで起動したケース（例: `env` 単体）は「抽出不能」と区別され、過剰拒否しない一方で Low には落とさない（`env` 単体は情報露出のため Medium 以上）。判別に確証がない場合は安全側に倒す。

### F-014: リスク判定対象の実体をハッシュ検証済みのものに結合する（セキュリティレビュー H-9/M-10）

リスク判定は、ハッシュ検証で確定した実体・パスに対して行わなければならない。現状は検証でハッシュを確定した後、リスク判定が独立にパス解決・ファイル情報の再取得を行うため、検証〜判定〜実行の間に **TOCTOU 窓** が開く。また検証済みハッシュは現状バイナリ解析の経路でのみ突合され、coreutils 判定・破壊判定には反映されていない。

- リスク判定が参照する実体・パスを、ハッシュ検証で確定したものに結合し、検証〜判定間の **独立した再解決・ファイル情報の再取得を最小化** する。やむを得ず再取得する場合は、確定済みハッシュと再突合する。
- さらに、確定した検証済み identity を **実際の実行（exec）時点まで束縛する**、または実行直前に再検証する。リスク判定後・exec 前の差し替え（別パスの実行）も検出・防止する（検証〜判定〜実行の全区間を対象とする）。
- 検証済みハッシュを持たないコマンドが、coreutils 判定・破壊判定など **バイナリ解析以外の経路だけで Low/通過しない** ようにする（F-005 の不確実扱いを、参照実体が検証済みハッシュを持たないあらゆる経路へ一貫適用する）。

**Acceptance Criteria**:
- **AC-64**: リスク判定（coreutils 判定の setuid チェックを含む）が参照する実体・パスが、ハッシュ検証で確定したものと一致することを保証する設計とし、検証〜判定間で差し替えられた場合に安全側（拒否）になる（または再 stat 時に content hash 不一致を検出する）。
- **AC-65**: 検証済みハッシュを持たないコマンドが、coreutils 判定・破壊判定のみで Low/通過とならず、F-005 の不確実扱い（実行中止）に一貫して帰着することをテストで検証する（解析無効構成 AC-51 と整合）。
- **AC-76**: 検証〜判定〜**実行（exec）** の全区間で同一の検証済み identity が使われることを保証する。リスク判定後・exec 前に対象が差し替えられた場合でも、別パス/別実体を実行しない（実行直前の再検証で検出する、または検証済み identity に束縛して exec する）。

### F-015: 任意コード実行コマンド（シェル・インタプリタ・ビルド/タスクランナー）を High に分類する（セキュリティレビュー）

以下のコマンドは、引数や外部定義ファイルを通じて **任意のシステムコール・任意コードを実行可能** であり、かつ実行される **実体（スクリプト・レシピ・ビルド定義）は本ツールのコマンド単位のリスク判定・ハッシュ検証の対象外** である。任意コード実行の能力に照らし、**実効リスクの High を下限** とする。

- **シェル**: `bash` / `sh` / `dash` / `zsh` 等（`-c` のインラインコード・スクリプトファイル・標準入力・REPL）
- **スクリプトインタプリタ・ランタイム**: `python` / `node` / `ruby` / `perl` / `php` / `lua` / `java` / `dotnet` / `pwsh` 等（既存プロファイルで AlwaysNetwork として扱われるもの）
- **ビルド/タスクランナー**: `make` / `cmake` / `ninja` / `gradle` / `mvn`（maven）/ `bazel` / `rake` / `just` / `task` 等。これらは `Makefile` 等の **検証されない外部定義に書かれた任意コマンドを実行** するため、シェル・インタプリタと同等のリスクを持つ。

現状シェル・インタプリタは「ネットワーク可能」を根拠に Medium と分類されるが過小評価である。バイナリ解析が exec syscall・動的ロード（`dlopen` 等）を High とする既存方針と整合させ、High を下限とする。これは F-013 のシェルインラインコード（`-c`）の High 下限（AC-61）を包含する。

注 1: `--version` / `--help` のような自明に無害なフラグのみの起動を除外するかは、無害な呼び出しの網羅的判別が困難で安全性とのトレードオフがあるため、**既定は一律 High（安全側）** とする。除外を導入する場合は、明示的なホワイトリストに限定し判別不能時は High に倒す方針でアーキテクチャ設計に委ねる（本要件では一律 High を確定する）。

注 2: 本リストは「任意コード実行を可能にする既知コマンド」の **キュレートされたブロックリスト** であり網羅的ではない（脅威モデル §1 のとおり、未知コマンドは Low 通過し得るため allowlist 併用が前提）。対象コマンドの具体リストと管理方針はアーキテクチャ設計で確定する。

**Acceptance Criteria**:
- **AC-73**: シェル・スクリプトインタプリタ（`bash` / `python` / `node` / `ruby` / `perl` 等）の実効リスクが、引数の有無・形態（スクリプトファイル / `-c` / REPL）によらず **High 以上** になる。
- **AC-74**: ビルド/タスクランナー（`make` / `cmake` / `gradle` 等）の実効リスクが **High 以上** になる。

## 3. 非機能要件

### NF-001: 後方互換性

- 既存の TOML 設定の解釈を破壊しない。ただし F-001/F-002/F-005/F-007/F-013/F-015 により、一部設定の挙動が変わる:
  - `claude` 等（実効リスク Medium→High）、`systemctl`/`service` の **変更系サブコマンド**（実行時 Low→High。読み取り専用サブコマンドは F-011 により Medium）、絶対パス指定の破壊的コマンド（Low→High）は、`risk_level` 設定によっては従来実行できていたものが拒否される。これは **意図的な安全性向上** である。
  - 任意コード実行コマンド（F-015）: シェル・インタプリタ（`bash`/`python`/`node`/`ruby`/`perl` 等）およびビルド/タスクランナー（`make`/`cmake`/`gradle` 等）の実効リスクが **High** になる（従来はシェル・インタプリタが Medium、ビルドツールはバイナリ解析依存）。これらを用いる自動化は `risk_level = "high"` の設定が必要になる（任意コード実行を意識的に許可させるため）。影響範囲が広いので移行ノートで強調する。
  - 間接実行（F-013）: ラッパー経由（`env`/`timeout`/`xargs` 等で包んだ）コマンドや、シェルのインラインコード実行（`bash -c "…"` 等）の実効リスクが上昇する（ラッパー経由特権昇格→Critical、シェル -c→High 下限 等）。従来これらを低い `risk_level` で実行していた設定は拒否され得る。回避手段は、ラップせず実コマンドを直接指定する、またはリスクに見合う `risk_level` を設定する。
  - バイナリ解析の不確実ケースは、従来 `risk_level = "high"` で実行できていたものが **設定によらず拒否** される。解消手段はケースごとに異なる: 解析レコード欠落・スキーマ不一致・ハッシュ不一致 → `record` の再実行による解析レコードの整備、バイナリの同一性が未検証 → ファイル検証の有効化（防御的分類であり、本番配線では通常発生しない。背景 E 参照）。
  - 解析無効構成（F-005 解析無効時）: 実行は常に拒否となる（従来は黙って通過）。dry-run は引き続き可能。オプトインによる実行許可は提供しない。
  - `risk_level = "unknown"` は設定エラーになる（従来は全コマンド拒否として黙って動作）。
- これらの変更点を移行ノートとしてドキュメントに記載する。

**Acceptance Criteria**:
- **AC-19**: 実効リスク上昇により拒否され得る代表的ケース（`claude` を `risk_level=medium` で実行、絶対パス `rm` を `risk_level=low` で実行、`systemctl` を `risk_level=low` または `medium` で実行、インタプリタ・シェル・ビルドツール `bash`/`python`/`make` 等を `risk_level=medium` 以下で実行、ラッパー/シェル経由実行、解析無効構成）、不確実ケースの設定によらない拒否（解消手段をケース別に記載）、および `"unknown"` の設定エラー化が、移行ノートまたは変更点としてドキュメントに記載される。

### NF-002: 実行時パスとdry-runパスの整合性

- 同一コマンドに対し、実行時の実効リスクと **dry-run に新設する実効リスク表示（F-009）** が一致する。dry-run の既存詳細表示との関係は F-009 本文のとおり（実行可否を決めるのは実効リスク側）であり、両表示の値が異なる場合の意味（詳細表示は参考情報）を出力またはドキュメントで明確にする。

**Acceptance Criteria**:
- **AC-20**: 既存の整合性テスト（`coreutils_consistency_test.go`）が修正後も成立し、破壊的コマンド（絶対パス含む）について両パスが同じ High を返すことを検証する。
- **AC-39**: プロファイル定義コマンドの代表例（`claude`, `systemctl`, `curl`）について、実行時とdry-runの実効リスクが一致することをテストで検証する。

### NF-003: 品質ゲート

- `make fmt` / `make test` / `make lint` がすべて成功する。

**Acceptance Criteria**:
- **AC-21**: 変更後、`make test` と `make lint` がパスする（既存の無関係な lint 指摘を除く）。

## 4. 用語

| 用語 | 定義 |
|------|------|
| 実効リスク（effective risk） | 実行時リスク判定がコマンド内容から算出する実際のリスクレベル |
| 最大許容リスク（maximum allowed risk） | `risk_level` 設定による実行可否の上限 |
| コマンドリスクプロファイル | コマンドに紐づくリスク定義。複数のリスク要因を保持し、その最大値を当該コマンドの基準リスクとする |
| エラー中止 | エラーを返してコマンド実行を中止する挙動（シンボリックリンク深度超過、予期しないレコード読込エラー等） |
| 無条件拒否（Critical 化） | 実効リスクを内部 Critical に昇格させ、`risk_level` 設定によらずリスク比較で拒否する挙動（F-005 の不確実ケース） |
| リスク昇格（許可可能） | **危険シグナル検出時**に安全側（より高いリスク、典型的には High）へ倒すが、`risk_level = "high"` で意識的に許可可能な挙動。**不確実ケースはこれに含めない**（不確実は「無条件拒否」側）。注: これは厳密には fail-closed ではない（許可可能であるため）。 |
| フェイルクローズド | 上記「エラー中止」「無条件拒否（Critical 化）」の総称（確実に実行させない挙動）。「リスク昇格（許可可能）」は含まない。 |
| 不確実ケース | バイナリの同一性または解析結果の信頼性を確認できない状態（解析レコード欠落、スキーマ/ハッシュ不一致、同一性未検証、解析エラー等）。F-005 により設定によらず実行中止の対象 |
| 単一の真実の源泉 | 同一コマンドのリスク定義を一箇所に集約し、各判定パスがそれを参照する設計方針 |
