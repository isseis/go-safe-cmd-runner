## 1. 概要 (Overview)

**目的:** このドキュメントは、Ubuntu 26.04 以降で Rust 化された coreutils 環境において、基本コマンドを安全に実行できるようにするための要件を定義する。

**背景:**
Ubuntu 26.04 から基本コマンド（coreutils）の多くが Rust で再実装され、`/usr/lib/cargo/bin/coreutils` 配下に **単一バイナリ（へのハードリンク群）** として配置されるようになった。`/usr/bin/mkdir` などの各コマンドはシンボリックリンクを辿るとこの単一バイナリに到達する。

この構成により、本ツールで次の 2 つの問題が発生している。

1. **リスク誤判定:** 単一バイナリには全サブコマンド共通の `dlsym` / `mprotect` / ソケット系シンボルが含まれるため、バイナリ解析で `mkdir` のような無害なコマンドまで HighRisk と判定され、実行がブロックされる。単一バイナリのバイナリ解析は、特定サブコマンドのリスク判定には原理的に役立たない。
2. **安全ディレクトリ判定の失敗:** パス名なしで TOML に記述したコマンドが、シンボリックリンク解決後に `/usr/lib/cargo/bin/coreutils/` 配下となり、安全なディレクトリに存在しないと判断される。

coreutils はシステム標準コマンドであり、`/usr/bin` などと同様に原則として安全なディレクトリとして扱いたい。また安全なコマンドについては、バイナリ解析結果によらずリスク判定を Low / Medium に下げたい。

## 2. 目的とスコープ (Goals and Scope)

### 2.1. 目的 (Goals)

-   Rust 化 coreutils 環境（単一バイナリ構成）で基本コマンドを安全に実行できるようにする。
-   coreutils ディレクトリをシステム標準の安全なディレクトリとして扱う。
-   単一バイナリのバイナリ解析結果に左右されない、適切なリスク判定を行う。

### 2.2. スコープ (In Scope)

-   `/usr/lib/cargo/bin/coreutils` 配下に解決されるコマンドの安全ディレクトリ判定。
-   `/usr/lib/cargo/bin/coreutils` 配下に解決されるコマンドのリスク判定。
-   実行時（ブロック判定）と dry-run（表示）の双方で一貫したリスク判定。

### 2.3. スコープ外 (Out of Scope)

-   `/usr/bin`、`/bin`、`/sbin` など coreutils 以外のディレクトリにおけるリスク判定方針の変更。
-   コマンド名を basename 化せず照合している既存処理（破壊的操作・システム変更判定）のフルパス対応全般の見直し（別タスクで扱う）。
-   coreutils 以外の Rust 化／単一バイナリ化されたツール群への対応。
-   coreutils ディレクトリパスの設定ファイルによる可変化。

## 3. 機能要件 (Functional Requirements)

-   **機能ID:** `F-001`
-   **機能名:** coreutils ディレクトリの安全ディレクトリ判定
-   **概要:** パス名なしで TOML に記述されたコマンドが coreutils ディレクトリ配下に解決された場合でも、安全なディレクトリにあると判定し実行を許可する。

-   **機能ID:** `F-002`
-   **機能名:** バイナリ解析に依存しないリスク判定
-   **概要:** coreutils ディレクトリ配下に解決されたコマンドは、単一バイナリのバイナリ解析結果に基づいて HighRisk と判定しない。

-   **機能ID:** `F-003`
-   **機能名:** コマンド別リスク分類
-   **概要:** coreutils 配下のコマンドのうち、既知の安全なコマンドは Low、それ以外のコマンドは Medium と判定する。安全なコマンドの一覧は本ツールが定義・保持する。

-   **機能ID:** `F-004`
-   **機能名:** 破壊的コマンドのリスク維持
-   **概要:** `rm`、`dd` などデータ消失やディスク破壊を伴う破壊的コマンドは、coreutils 化後も従来同様 High のリスクとして扱う（安全コマンドとして降格しない）。

-   **機能ID:** `F-005`
-   **機能名:** 実行時と dry-run の一貫性
-   **概要:** 同一の coreutils コマンドに対して、dry-run で表示されるリスクと実行時のブロック判定に用いるリスクが一致する。

## 4. 非機能要件 (Non-Functional Requirements)

### 4.1. セキュリティ (Security)
-   本対応の適用範囲を coreutils ディレクトリ配下に限定し、それ以外のディレクトリにおける既存のリスク検出能力を低下させないこと。
-   特権昇格（`sudo` 等）の検出は本対応の影響を受けず、従来どおり Critical として扱われること。
-   coreutils 配下の安全コマンド一覧に含まれない（未知の）コマンドは、安全側に倒して Medium 以上で扱うこと。

### 4.2. 互換性 (Compatibility)
-   coreutils が Rust 化されていない従来の Linux 環境における挙動を変えないこと。
-   既存のリスクレベル体系（Low / Medium / High / Critical）に従うこと。

### 4.3. 保守性 (Maintainability)
-   coreutils コマンドのリスク分類（安全コマンド一覧等）を追加・変更しやすい形で保持すること。

## 5. 制約条件 (Constraints)

-   coreutils ディレクトリのパスは `/usr/lib/cargo/bin/coreutils` とする。
-   既存のリスク評価の枠組み（特権昇格・破壊的操作・ネットワーク操作・システム変更・既定リスク）と整合させること。
-   使用言語は Go とする。

## 6. 用語集 (Glossary)

-   **coreutils 単一バイナリ:** Ubuntu 26.04 以降で Rust 実装された coreutils 群が `/usr/lib/cargo/bin/coreutils` 配下に持つ、全サブコマンド共通の単一実行ファイル（およびそれへのハードリンク群）。
-   **安全なディレクトリ:** 本ツールがシステム標準として信頼し、配下のコマンド実行を原則許可するディレクトリ（例：`/usr/bin`、`/bin`、`/sbin`）。
-   **バイナリ解析:** 実行ファイルの ELF シンボルや syscall を解析してコマンドのリスクを推定する処理。
-   **リスクレベル:** コマンドの危険度を表す段階（Low / Medium / High / Critical）。設定された許容レベルを超える場合に実行をブロックする。
-   **dry-run:** 実際にコマンドを実行せず、リスク判定や実行計画を表示する動作モード。
