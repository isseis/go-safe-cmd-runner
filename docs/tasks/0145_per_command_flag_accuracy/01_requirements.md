> 抽出層の構造のみを刷新し観測可能な挙動を一切変えなかった。本タスクはその上で、各コマンドの

## 1. 背景と目的

### 1.1 背景

0142（判断軸2: 宛先パス信頼区分）はファイル操作コマンドのオペランド抽出を実装し、各コマンドが独自に
フラグ集合を保持していた。0144 はこれを宣言的フラグ仕様表＋単一 getopt パーサへ集約したが、**挙動保存**の
原則上、現行のフラグ集合をそのまま写し取った。その過程で、現行の一部コマンドが**実 CLI に存在しないフラグを
真偽フラグとして認識している**（過剰認識）ことが、0144 のレビューで具体的に指摘された。

代表例（0142 由来。共有ヘルパ `extractSimpleWrite` の汎用真偽フラグ集合に起因）:

- `sponge`: 実 CLI は `-a` のみだが、現行は `-c`/`-h`/`-p`/`-v`/`-f`/`-i`/`-r` も真偽フラグとして認識する。
- `mkdir`: 実 CLI に無い `-a`/`-c`/`-h`/`-f`/`-i`/`-r` を真偽フラグとして認識する。
- `touch`: 実 CLI に無い `-p`/`-v`/`-i` を真偽フラグとして認識する。

これらは「コマンドが実際には持たないフラグを既知として受理する」過剰認識であり、本来は未知フラグとして
`recognized=false`（fail-closed）に倒すべき入力を `recognized=true` にしている。

### 1.2 セキュリティ上の位置づけ（なぜ緊急ではないか）

本件は**精度（モデルの正確さ）の問題**であり、実害のある fail-open ではない。理由:

- 過剰認識されているのはすべて**真偽フラグ（値を取らない）**であり、非フラグ引数（path）を吸収しない。
  よって「引数付きフラグの値（path）の取りこぼし」= fail-open は生じない。
- 判断軸2 のセキュリティ保証は抽出精度ではなく fail-closed な `Recognized` contract が担う
  （[0142/02 §3.2] を継承）。ただし「安全側へ縮退する」が成り立つのは精度不足のうち**宣言漏れ**
  （本来受理すべきフラグを未知扱いし `recognized=false` にする＝過剰分類）の場合である。**過剰認識**
  （本来却下すべき入力を受理し `recognized=true` にする）は fail-closed には倒れない。ただし上の項のとおり
  過剰認識は真偽フラグに限られ path を取りこぼさないため、fail-open（実害）も生じない。よって過剰認識は
  安全性の欠陥ではなく精度の問題であり、本タスクで是正する。

したがって本タスクは安全性の修復ではなく、**モデルの正確性と保守性の向上**を目的とする。0144 で
フラグ知識がデータ化されたため、是正は仕様データ 1 箇所の編集で完結できる（0144 の NF-002／§9 が想定する拡張）。

### 1.3 ゴール

- 各対象コマンドの宣言的フラグ仕様（`FlagSpec` 群）を、実 CLI のフラグ集合・表記・アリティ・値の役割に整合させる。
- 過剰認識（実 CLI に無いフラグの受理）と宣言漏れ（実 CLI にあるフラグの未宣言）を解消する。
- 挙動変化（`recognized` 等）はすべて安全側であることを確認し、意図的逸脱として記録・テストで明示する。

## 2. スコープ

- **In**:
  - **F-001**: 対象コマンドのフラグ仕様の実 CLI 整合（過剰認識の除去・宣言漏れの追加・表記/アリティ/値役割の修正）。
  - **F-002**: 挙動変化の管理（安全側であることの確認と、差分テストにおける意図的逸脱としての明示・記録）。
  - **F-003**: 完全性メタテストと既存テストの維持（0144 のメタテストは緑を保ち、意図的に変える既存テスト期待値のみ
    根拠付きで更新する）。
- **Out**:
  - 判断軸2 のレベル規則・区分判定・操作固有の下限の変更（0142 で確定。本タスクは抽出するフラグ集合のみ整合させ、
    区分判定・リスクレベル合成・リゾルバには触れない）。
  - 単一 getopt パーサ（0144 `parseArgs`）の文法・挙動変更（パーサは不変。仕様データのみ変える）。
  - 新規ファイル操作コマンドの追加（既存コマンドの整合に閉じる）。
  - 値内文法ヘルパ（`chmodGrantsHigh`/`aclGrantsWrite`/`tarMode` 等）の変更（不変）。

## 3. 機能要件と受け入れ基準

### F-001: コマンド別フラグ仕様の実 CLI 整合

各対象コマンドの `FlagSpec` 群を実 CLI に合わせる。過剰認識フラグを除去し、宣言漏れフラグを追加し、各フラグの
表記（短縮形・長形式）・アリティ（真偽/引数必須/引数省略可）・値の役割（write/read/非 path）を実 CLI に整合させる。
共有ヘルパ由来の汎用集合（`simpleWriteFlags` 等）に起因する不正確さを解消する。

**Acceptance Criteria**:
- **AC-01**: 対象各コマンドの宣言フラグ集合が実 CLI のフラグ集合と一致する（実 CLI に無いフラグを宣言しない・
  実 CLI にあるフラグの宣言漏れがない）。検証は各コマンドの権威ある仕様（GNU coreutils 等の man ページ）を典拠とし、
  典拠を実装計画に明記する。
- **AC-02**: 少なくとも §1.1 で挙げた `sponge`・`mkdir`・`touch` の過剰認識フラグが除去され、当該フラグを与えた
  入力（例 `sponge -r FILE`、`mkdir -a DIR`、`touch -p FILE`）が、現行の `recognized=true` から実 CLI 整合後の
  期待値（未知フラグとして `recognized=false`）へ変わることがテストで表明される。
- **AC-03**: 各フラグの表記・アリティ・値の役割が実 CLI と一致する（短縮形/長形式の網羅、引数省略可の正確な区別を含む）。
  0144 の完全性メタテスト（引数付きフラグは具体的な `ValueRole` を持つ）は引き続き満たす。

### F-002: 挙動変化の安全性確認と意図的逸脱の明示

本タスクは挙動を変える。変える各入力について安全側であることを確認し、差分テストの意図的逸脱として記録する。

**Acceptance Criteria**:
- **AC-04**: フラグ整合により `LocationResult`（`Recognized`/`Level`/`Operands`/`ReasonCodes` 等）が変わる入力が、
  0144 の差分テスト基盤（`extraction_diff_test.go` の `diffExclusions`）に**意図的逸脱として登録**され、各逸脱に
  「どのコマンドのどのフラグで何がどう変わるか」の理由がコメントで明記される。逸脱は当該コマンド×当該入力形に
  厳密一致させ、無関係な入力を巻き込まない。
- **AC-05**: すべての挙動変化が安全側であることを確認・記録する。すなわち (a) 過剰認識の除去は `recognized=true→false`
  （過剰分類＝fail-closed 方向）であり、(b) 新たな fail-open（引数付きフラグの path 値の取りこぼし、または捕捉済み
  path 値の無視）を生じない。

### F-003: 完全性・回帰の維持

**Acceptance Criteria**:
- **AC-06**: 0144 の完全性メタテスト・アリティ不変条件・仕様カバレッジテスト（`flag_spec_test.go`）が緑である。
- **AC-07**: 既存の挙動テスト（`destination_zoning_test.go`・`operand_path_resolver_test.go`）が、本タスクで意図的に
  変える分を除き緑である。期待値を変更する場合は、その入力が F-002 の安全側挙動変化に該当することを根拠として
  実装計画に記録する（無根拠の期待値変更は不適合）。

## 4. 非機能要件

- **NF-001**: `make fmt`・`make test`・`make lint` が成功する。
- **NF-002**: 保守性: 是正は宣言的仕様データの編集で完結し、パーサや区分判定のコード分岐追加を要さない
  （0144 の NF-002 を継承）。
- **NF-003**: 決定性・read-only を維持する（live identity 不参照・副作用なし。0144 の NF-003 を継承）。
- **NF-004**: 前提依存: 0144（宣言的フラグ仕様化）がマージ済みであること。本タスクは 0144 の `commandFlagSpecs`・
  差分テスト基盤・完全性メタテストの上に成り立つ。

## 5. スコープ外の根拠

- 本タスクは 0144 が**挙動保存**のため意図的に先送りした「精度改善」を扱う（0144/01 §5 が別タスクへ切り出した内容）。
  0142 のレベル規則・区分・下限・リゾルバは確定済みで、本タスクはそれらの意味を変えない（変えるのは抽出する
  フラグ集合の正確さのみ）。
- パーサ（0144 `parseArgs`）は不変。フラグ知識はデータ側で完結させるという 0144 の設計を踏襲する。

## 6. 成功基準（要約）

- AC-01〜AC-07 が test/static で緑。
- §1.1 の代表的な過剰認識（`sponge`/`mkdir`/`touch`）が解消され、対象コマンド全体のフラグ仕様が実 CLI に整合する。
- すべての挙動変化が安全側であることが差分テストの意図的逸脱として明示・記録されている。
