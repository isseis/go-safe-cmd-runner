# ライブラリ解析結果の共通キャッシュ化 要件定義書

## 1. 概要

### 1.1 背景

タスク 0123 では、実行ファイルの `DynLibDeps` を辿ってライブラリ単位の解析を行い、
結果を実行ファイルごとの `fileanalysis.Record` に記録する設計となっている。

この方式は機能要件を満たす一方で、以下の課題がある。

- 同一ライブラリ（例: `libssl.so.3`）の解析結果が複数の実行ファイルレコードに重複保存される
- レコード JSON の肥大化により、ディスク使用量と I/O コストが増加する
- 解析対象ライブラリが更新された場合の整合性管理がレコード単位に分散する

一方で、既存の `libc-cache` は「ライブラリを一度だけ解析し、ハッシュで同一性を検証して再利用する」方式を採用しており、
同様のアプローチをライブラリ解析全体へ拡張することで、重複解析・重複保存を抑制できる。

### 1.2 目的

ライブラリ解析結果を実行ファイルごとのレコードへ都度埋め込む方式から、
**ライブラリ単位の共通キャッシュ方式**へリファクタリングし、以下を実現する。

1. 同一ライブラリの解析を 1 回で済ませ、結果を再利用する
2. 解析時と利用時の同一性を `lib_hash` で担保する
3. 実行ファイルレコードは軽量な DynLibDep 中心にする
4. 動的ロードシンボル（dlopen等）を含むライブラリも正しく検出し、リスク判定に反映可能とする
5. 既存のネットワーク判定精度（AC 互換）を維持する

### 1.3 スコープ

- 対象: Linux ELF 向けライブラリ解析結果（タスク 0123 で追加した解析経路）
- 対象外: Mach-O 専用キャッシュの統合（既存 `libSystem` キャッシュは現状維持）
- 対象外: 関数単位到達可能性解析（タスク 0123 の制約を維持）

---

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| Library Analysis Cache | ライブラリ単位の解析結果を保存する共通キャッシュ |
| Cache Entry | 1 つのライブラリファイル（`lib_path` + `lib_hash`）に対応するキャッシュレコード |
| DynLibDep | 実行ファイルレコードが保持する動的リンクライブラリ依存情報（soname / path / hash） |
| Dynamic Load Signal | `dlopen`, `dlsym`, `dlvsym` シンボルの検出結果 |
| Cache Hit | `lib_hash` 一致により既存エントリを再利用できる状態 |
| Cache Miss | エントリが存在しない、または `lib_hash` 不一致で再解析が必要な状態 |

## 3. 機能要件

### 3.1 キャッシュデータモデル

#### FR-3.1.1: ライブラリ解析キャッシュファイル

ライブラリ解析結果は、実行ファイルレコードとは別にキャッシュファイルとして保存する。
既存の `libc-cache`（`libccache` パッケージ）とは用途・スキーマ・保存場所をすべて分離する。

- `libc-cache`: syscall 番号 → ネットワーク判定テーブル。`record` が読む付加情報
- ライブラリ解析キャッシュ（本タスク）: 各共有ライブラリの syscall・dlopen シグナル解析結果。`record` が書き、`runner` が読む

必須フィールド:

- `schema_version`
- `lib_path`（解析対象ライブラリの絶対パス）
- `lib_hash`（`sha256:<hex>`）
- `analyzed_at`（RFC3339）
- `syscall_analysis`（任意）
- `symbol_analysis`（任意）
- `has_dynamic_load_signal`（ライブラリ単体が動的ロードシンボルを持つか）
- `has_network_signal`（ライブラリ単体がネットワークシグナルを持つか）

#### FR-3.1.2: キャッシュキーと同一性担保

キャッシュ再利用判定は少なくとも以下で行う。

1. `lib_path` が一致
2. `lib_hash` が一致
3. `schema_version` が一致

いずれか不一致の場合は Cache Miss とし、再解析してキャッシュを更新する。

### 3.2 実行ファイルレコードの保持方式

#### FR-3.2.1: 実行ファイルレコードは参照中心

`fileanalysis.Record` にはライブラリ解析結果の全量を埋め込まず、
動的リンクライブラリ依存情報（DynLibDep）を保存する。

DynLibDep の必須情報:

- `soname`
- `path`
- `hash`

#### FR-3.2.2: runner による実行時キャッシュ参照

`runner` は実行時に以下の順序でリスク判定を行う。

1. 実行ファイルレコード（JSON）を読み込み、`dyn_lib_deps` の一覧を取得する
2. 各ライブラリについて `lib_path` + `lib_hash` に対応するキャッシュファイルを読み込む
   （runner はコマンド実行時にすべての共有ライブラリを開いてハッシュを検証するため、
   この時点でライブラリファイルへのアクセスとハッシュ取得は済んでいる）
3. キャッシュファイルの `has_network_signal` / `has_dynamic_load_signal` を集計し、
   ネットワーク利用および高リスク判定を行う

キャッシュファイルが存在しない場合、またはキャッシュの `lib_hash` が
実行時のハッシュと一致しない場合はエラーとして処理を停止する。

### 3.3 record フロー

#### FR-3.3.1: キャッシュ優先フロー

`record` 実行時、各ライブラリについて以下順で処理する。

1. `DynLibDeps` から対象ライブラリを抽出（0123 の除外規則を継承）
2. Cache Lookup（`path/hash/schema`）
3. Hit: キャッシュ結果を再利用
4. Miss: 解析実行しキャッシュ保存
5. 実行ファイルレコードに DynLibDep を保存

#### FR-3.3.2: 同一実行内キャッシュ

既存のセッション内メモリキャッシュ（同一 `record` 実行中の重複回避）は維持する。

- 優先順位: メモリキャッシュ → 永続キャッシュ → 再解析

### 3.4 エラー処理

#### FR-3.4.1: キャッシュ読込エラー

キャッシュファイル読込・パースに失敗した場合は警告を記録し、当該ライブラリを再解析する。

#### FR-3.4.2: キャッシュ保存エラー

キャッシュ保存に失敗した場合でも、実行ファイル解析結果の保存は継続する。

- 失敗は `AnalysisWarnings` に追記する

なお、ライブラリ解析キャッシュが保存されなかったライブラリについては、
`runner` 実行時にキャッシュミスエラーが発生する（FR-3.2.2 参照）。

### 3.5 既存機能との整合

#### FR-3.5.1: 0123 の除外規則継承

以下は継承して維持する。

- syscall wrapper ライブラリ除外
- VDSO 除外
- `ErrUnsupportedArch` の扱い
- 非致命エラーを `AnalysisWarnings` に記録して継続

#### FR-3.5.2: 既存 runner 判定との互換

ライブラリ解析キャッシュの `has_network_signal` を用いた runner ネットワーク判定は、
0123 の `DetectedLibraryNetworkDeps` ベースの判定と同等の結果を返す（後退なし）。

## 4. 非機能要件

### 4.1 パフォーマンス

- 同一ライブラリを複数実行ファイルが参照するケースで、再解析回数を有意に削減する
- 実行ファイルレコード JSON サイズを、0123 の埋め込み方式より削減する

### 4.2 スキーマ

- `fileanalysis.Record` のスキーマ変更が必要な場合は `CurrentSchemaVersion` を更新する
- ライブラリ解析キャッシュにも独立した `schema_version` を持たせる

### 4.3 可観測性

`record` 実行ログに以下を出力可能とする。

- cache hit / miss
- 再解析理由（hash mismatch, schema mismatch, cache corruption など）

## 5. 受け入れ基準

| ID | 基準 |
|----|------|
| AC-1 | 同一ライブラリを複数実行ファイルが参照する場合、永続キャッシュにより 2 回目以降の解析がスキップされる |
| AC-2 | ライブラリファイル内容が更新され `lib_hash` が変わった場合、キャッシュミスとなり再解析される |
| AC-3 | キャッシュヒット時でも実行ファイルレコードに正しい DynLibDep（soname / path / hash）が記録される |
| AC-4 | `runner` がライブラリ解析キャッシュを参照してネットワーク判定した結果が、0123 実装時と同等（後退なし）である |
| AC-5 | `record` 時にキャッシュファイルが壊れている場合、警告記録のうえ再解析して処理継続できる |
| AC-6 | 実行ファイルレコードのサイズが、ライブラリ解析全量埋め込み方式より削減されることを確認できる |
| AC-7 | syscall wrapper / VDSO 除外ルールがキャッシュ方式でも維持される |
| AC-8 | `make fmt` / `go test -tags test -v ./...` / `make lint` がすべて成功する |
| AC-9 | `dlopen`/`dlsym` 等のシンボルを持つライブラリのキャッシュファイルで `has_dynamic_load_signal` が `true` となり、`runner` による高リスク判定が機能する |

---

## 6. 制約事項

1. キャッシュはハッシュ一致を前提とした再利用であり、実行時に `dlopen` でロードされるライブラリの追跡・解析はスコープ外（ライブラリ自身が `dlopen`/`dlsym` 等のシンボルを参照するかの検出はスコープ内）
2. ライブラリキャッシュ導入後も、初回解析コスト自体は 0123 と同等
3. キャッシュ破損時は安全側（再解析）へフォールバックする

---

## 7. 段階的移行方針（推奨）

1. キャッシュマネージャ実装（読み書き・整合性判定）
2. `record` の library 解析フローをキャッシュ優先へ差し替え
3. `Record` を参照中心モデルへ移行（必要に応じてスキーマ更新）
4. 回帰テストで 0123 の判定互換を確認

この順序により、機能回帰リスクを抑えて段階的に移行できる。
