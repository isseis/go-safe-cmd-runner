# 動的ライブラリ解析結果ストア導入 要件定義書

## 1. 概要

### 1.1 背景

タスク 0123 では、実行ファイルの DynLibDeps を辿ってライブラリ単位の解析を行い、
結果を実行ファイルごとの fileanalysis.Record に記録する設計となっている。

この方式は機能要件を満たす一方で、以下の課題がある。

- 同一ライブラリ（例: libssl.so.3）の解析結果が複数の実行ファイルレコードに重複保存される
- レコード JSON の肥大化により、ディスク使用量と I/O コストが増加する
- 解析対象ライブラリが更新された場合の整合性管理がレコード単位に分散する

既存の libc-cache は、再解析回避のために解析成果物を保存・再利用する方式を採用している。
同様の方式をライブラリ解析全体に拡張することで、重複解析と重複保存を抑制できる。

### 1.2 目的

ライブラリ解析結果を実行ファイルレコードへ都度埋め込む方式から、
動的ライブラリ解析結果をライブラリ単位で保存する方式へ移行し、以下を実現する。

1. 同一ライブラリの解析を 1 回で済ませ、結果を再利用する
2. 解析時と利用時の同一性を lib_hash で担保する
3. 実行ファイルレコードは軽量な DynLibDep 中心にする
4. 動的ロードシンボル（dlopen 等）を含むライブラリも正しく検出し、リスク判定に反映可能とする
5. 既存のネットワーク判定精度（AC 互換）を維持する

### 1.3 スコープ

- 対象: Linux ELF 向け動的ライブラリ解析結果（タスク 0123 で追加した解析経路）
- 対象外: Mach-O 専用キャッシュの統合（既存 libSystem キャッシュは現状維持）
- 対象外: 関数単位到達可能性解析（タスク 0123 の制約を維持）

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| Dynamic Library Analysis Result | 1 つの共有ライブラリから抽出された syscall_analysis / symbol_analysis / dynamic_load_symbols |
| Dynamic Library Analysis Store | Dynamic Library Analysis Result を保存・取得する永続ストア |
| Reanalysis Avoidance Cache | record が同一ライブラリ再解析を避けるためにストアを利用する運用観点 |
| Analysis Entry | 1 つのライブラリファイル（lib_path + lib_hash）に対応する解析結果レコード |
| DynLibDep | 実行ファイルレコードが保持する動的リンクライブラリ依存情報（soname / path / hash） |
| Analysis Reuse Hit | 同一性判定が一致し、既存の解析結果を再利用できる状態 |
| Analysis Result Not Found | 解析結果が存在しない、または同一性不一致で再解析が必要な状態 |

**用語規則:**

- 「キャッシュ」は record の再解析回避という運用観点でのみ用いる
- runner の判定対象は常に「動的ライブラリ解析結果」と記載する

## 3. 機能要件

### 3.1 データモデル

#### FR-3.1.1: 動的ライブラリ解析結果ファイル

動的ライブラリ解析結果は、実行ファイルレコードとは別に解析結果ファイルとして保存する。
libc-cache（libccache パッケージ）とは用途・スキーマ・保存場所を分離する。

- libc-cache: syscall 番号をネットワーク判定に対応付ける補助情報
- 動的ライブラリ解析結果ストア（本タスク）: 共有ライブラリ単位の解析結果。record が書き、runner が読む

必須フィールド:

- schema_version
- lib_path（解析対象ライブラリの絶対パス）
- lib_hash（sha256:<hex>）
- syscall_analysis（ライブラリ内で検出したシステムコール呼び出し情報）
- symbol_analysis（ライブラリ内で検出したシステムコールラッパ関数シンボル情報）
- dynamic_load_symbols（ライブラリ内で検出した動的ロード関連シンボル名の列挙）

補足:

- 解析結果スキーマは、実行ファイル（コマンド）の解析結果スキーマと整合する形で定義する
- has_dynamic_load_signal / has_network_signal は保存せず、runner が実行時に導出する

#### FR-3.1.2: 解析結果同一性判定

再利用判定は少なくとも以下で行う。

1. lib_path が一致
2. lib_hash が一致
3. schema_version が一致

いずれか不一致の場合は Analysis Result Not Found とし、record 時は再解析して保存する。

### 3.2 実行ファイルレコードの保持方式

#### FR-3.2.1: 実行ファイルレコードは参照中心

fileanalysis.Record にはライブラリ解析結果の全量を埋め込まず、
動的リンクライブラリ依存情報（DynLibDep）を保存する。

DynLibDep の必須情報:

- soname
- path
- hash

#### FR-3.2.2: runner による実行時解析結果参照

runner は実行時に以下の順序でリスク判定を行う。

1. 実行ファイルレコード（JSON）を読み込み、dyn_lib_deps の一覧を取得する
2. 各ライブラリについて lib_path + lib_hash に対応する解析結果を取得する
   （runner はコマンド実行時にすべての共有ライブラリを開いてハッシュを検証するため、
   この時点でライブラリファイルへのアクセスとハッシュ取得は済んでいる）
3. 解析結果の syscall_analysis / symbol_analysis / dynamic_load_symbols を参照し、
   runner が has_network_signal / has_dynamic_load_signal を導出して集計し、
   ネットワーク利用および高リスク判定を行う

解析結果が取得できない場合、または解析結果の lib_hash が実行時ハッシュと一致しない場合は
エラーとして処理を停止する。

### 3.3 record フロー

#### FR-3.3.1: 解析結果優先フロー

record 実行時、各ライブラリについて以下順で処理する。

1. DynLibDeps から対象ライブラリを抽出（0123 の除外規則を継承）
2. 解析結果 Lookup（path/hash/schema）
3. Analysis Reuse Hit: 既存解析結果を再利用
4. Analysis Result Not Found: 解析実行し解析結果を保存
5. 実行ファイルレコードに DynLibDep を保存

#### FR-3.3.2: 同一実行内キャッシュ

同一 record 実行中の重複回避は維持する。

- 優先順位: メモリキャッシュ → 永続ストア → 再解析

### 3.4 エラー処理

#### FR-3.4.1: 解析結果読込エラー

解析結果ファイル読込・パースに失敗した場合は警告を記録し、当該ライブラリを再解析する。

#### FR-3.4.2: 解析結果保存エラー

解析結果保存に失敗した場合でも、実行ファイル解析結果の保存は継続する。

- 失敗は AnalysisWarnings に追記する

なお、解析結果が保存されなかったライブラリについては、runner 実行時に
解析結果未取得エラーが発生する（FR-3.2.2）。

### 3.5 既存機能との整合

#### FR-3.5.1: 0123 の除外規則継承

以下は継承して維持する。

- syscall wrapper ライブラリ除外
- VDSO 除外
- ErrUnsupportedArch の扱い
- 非致命エラーを AnalysisWarnings に記録して継続

#### FR-3.5.2: 既存 runner 判定との互換

動的ライブラリ解析結果（syscall_analysis / symbol_analysis / dynamic_load_symbols）を
用いた runner 判定は、0123 の DetectedLibraryNetworkDeps ベース判定と同等の結果を返す。

### 3.6 タスク 0123 テストカバレッジの引き継ぎ

タスク 0123 の整合性チェックにより、以下のテストが未作成であることが判明した。
本タスクのリファクタリングで analyzeLibraries() および analyzeOneLibrary() の
実装経路が変わるため、これらのテストを本タスクで追加する。

#### FR-3.6.1: ファイルサイズ上限超過時のエラー処理

Analyze() がファイルサイズ超過（1 GB 超）を検出した場合、エラーとして扱い、
ファイル不在（FR-3.6.2）と同様の処理を行う。

背景: サイズ超過でスキャンをスキップした場合、機械語中の直接 syscall 命令が検出できず、
「ネットワーク利用なし」と誤記録するリスクが生じる。セキュリティツールとして false negative は
許容しがたいため、エラーとして扱い不完全な解析結果を記録しない。

- 訂正: タスク 0123 AC-10 の warning + スキップ継続設計を取り消し、
  analyzeOneLibrary は error を返す動作へ変更する

#### FR-3.6.2: ライブラリファイル不在時のエラー処理

DynLibDeps に記録されたライブラリファイルが analyzeLibraries() 時点で存在しない場合、
その実行ファイルの解析はエラーとして扱い、当該レコードは書き込まない。

背景: analyzeDynLibDeps() でパスとハッシュを確認した直後に analyzeLibraries() を実行するため、
ファイル不在は競合状態や実装バグ等の予期しない状態を示す。不完全な解析結果を
ネットワーク利用なしと誤記録するリスクを避けるため、エラーとして扱う。

record は複数ファイルをバッチ処理するため、ある実行ファイルの解析がエラーでも
次ファイル処理は継続する（ファイル単位エラー）。

- 訂正: タスク 0123 AC-7 の warning + 継続設計を取り消し、
  analyzeOneLibrary は error を返す動作へ変更する

#### FR-3.6.3: VDSO 除外の専用テスト

VDSO（linux-vdso.so.1 等）のみが DynLibDeps に含まれる場合も
analyzeLibraries() の解析対象から除外されること。

## 4. 非機能要件

### 4.1 パフォーマンス

- 同一ライブラリを複数実行ファイルが参照するケースで、再解析回数を有意に削減する
- 実行ファイルレコード JSON サイズを、0123 の埋め込み方式より削減する

### 4.2 スキーマ

- fileanalysis.Record のスキーマ変更が必要な場合は CurrentSchemaVersion を更新する
- 動的ライブラリ解析結果ファイルにも独立した schema_version を持たせる

### 4.3 可観測性

record 実行ログに以下を出力可能とする。

- analysis reuse hit / result not found
- 再解析理由（hash mismatch, schema mismatch, parse error など）

## 5. 受け入れ基準

| ID | 基準 |
|----|------|
| AC-1 | 同一ライブラリを複数実行ファイルが参照する場合、永続ストア再利用により 2 回目以降の解析がスキップされる |
| AC-2 | ライブラリファイル内容が更新され lib_hash が変わった場合、解析結果未取得扱いとなり再解析される |
| AC-3 | 解析結果再利用時でも実行ファイルレコードに正しい DynLibDep（soname / path / hash）が記録される |
| AC-4 | runner が動的ライブラリ解析結果を参照して判定した結果が、0123 実装時と同等（後退なし）である |
| AC-5 | record 時に解析結果ファイルが壊れている場合、警告記録のうえ再解析して処理継続できる |
| AC-6 | 実行ファイルレコードのサイズが、ライブラリ解析全量埋め込み方式より削減されることを確認できる |
| AC-7 | syscall wrapper / VDSO 除外ルールが維持される |
| AC-8 | make fmt / go test -tags test -v ./... / make lint がすべて成功する |
| AC-9 | dlopen / dlsym 等のシンボルを持つライブラリの解析結果に dynamic_load_symbols が記録され、runner が内部導出した has_dynamic_load_signal により高リスク判定が機能する |
| AC-10 | ファイルサイズが 1 GB を超えるライブラリの Analyze() 呼び出しがエラーを返し、analyzeLibraries() がそのエラーを上位へ伝播する。当該実行ファイルのレコードは書き込まれず、record セッションは次ファイル処理を継続する（FR-3.6.1） |
| AC-12 | 存在しないライブラリパスが analyzeLibraries() に渡された場合、エラーが返されその実行ファイルのレコードは書き込まれない。record セッションは次ファイル処理を継続する（FR-3.6.2） |
| AC-13 | linux-vdso.so.1 等の VDSO のみが DynLibDeps に含まれる場合、analyzeLibraries の解析対象から除外される専用テストが存在する（FR-3.6.3） |

---

## 6. 制約事項

1. 解析結果再利用はハッシュ一致を前提とし、実行時に dlopen でロードされる追加ライブラリの追跡はスコープ外
2. 解析結果ストア導入後も、初回解析コスト自体は 0123 と同等
3. 解析結果破損時は安全側（再解析）へフォールバックする

---

## 7. 段階的移行方針（推奨）

1. 解析結果ストア実装（読み書き・整合性判定）
2. record の library 解析フローを解析結果優先へ差し替え
3. Record を参照中心モデルへ移行（必要に応じてスキーマ更新）
4. 回帰テストで 0123 判定互換を確認

この順序により、機能回帰リスクを抑えて段階的に移行できる。
