# Mach-O `LC_LOAD_DYLIB` 整合性検証 要件定義書

## 1. 概要

### 1.1 背景

タスク 0073 では Mach-O バイナリのインポートシンボルを解析し、ネットワーク関連シンボルを
検出する機能を実装した。しかし以下のケースには未対応である：

1. **依存 `.dylib` の差し替え（サプライチェーン攻撃）**: バイナリが `LC_LOAD_DYLIB` で明示的に
   リンクしている `.dylib` が `record` 時から差し替えられた場合、現行の実装では検出できない。

2. **言語ランタイム（`libruby`, `libpython` 等）経由のネットワーク利用**: バイナリの
   インポートシンボルにはネットワーク API が現れないが、依存 `.dylib` がネットワーク API を
   提供・呼び出すケース。

これらのリスクに対応するため、以下の機能拡張を行う：

- **方策 A**: `record` 時に Mach-O バイナリの `LC_LOAD_DYLIB` / `LC_LOAD_WEAK_DYLIB`
  から依存ライブラリを解決し、各ライブラリのフルパスとハッシュを記録する。`runner` 実行時に
  現在のライブラリのハッシュと照合し、差し替えを検出する。
- **方策 B（スコープ外）**: `LC_LOAD_DYLIB` ベースの既知ネットワークライブラリ検出は
  タスク 0098 で扱う。

本タスクは ELF 版のタスク 0074（`DT_NEEDED` 整合性検証）の Mach-O 対応版であり、
タスク 0095 FR-4.3 の実装を担う。

### 1.2 目的

- `LC_LOAD_DYLIB` / `LC_LOAD_WEAK_DYLIB` で参照される依存ライブラリの差し替えを検出する
- `record` 時と `runner` 実行時の依存ライブラリの一貫性を保証する
- dyld shared cache 内のシステムライブラリ（`libSystem.B.dylib` 等）については、
  ファイルシステム上に実体が存在しない場合を適切に扱う

### 1.3 スコープ

- **対象**: macOS (Darwin) 上の Mach-O バイナリ（単一アーキテクチャおよび Fat バイナリ）
- **対象**: `LC_LOAD_DYLIB` および `LC_LOAD_WEAK_DYLIB` に列挙される依存ライブラリ
- **対象**: 間接依存ライブラリの再帰的解決（BFS、FR-3.1.9 参照）
- **対象外**: `dlopen()` による実行時ロード（タスク 0074 方策 B で既に対応済み）
- **対象外**: dyld shared cache からの `.dylib` 抽出によるハッシュ検証（タスク 0100 の扱い）
- **対象外**: `DYLD_INSERT_LIBRARIES` による注入攻撃（`runner` 側で環境変数クリア済み）
- **対象外**: macOS コード署名検証（本システムのハッシュ検証と相互補完として独立して運用）
- **対象外**: iOS / iPadOS / tvOS / watchOS 向けバイナリ
- **対象外**: スクリプトファイル・非 Mach-O ファイル

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| `LC_LOAD_DYLIB` | Mach-O Load Command の一種。バイナリが実行時に必ずリンクする共有ライブラリ（`.dylib`）を指定する。ELF の `DT_NEEDED` に相当 |
| `LC_LOAD_WEAK_DYLIB` | `LC_LOAD_DYLIB` の弱リンク版。ライブラリが存在しない場合でもバイナリはロード可能。存在しない場合の解決失敗はエラーとしない |
| `LC_RPATH` | Mach-O におけるランタイム検索パスエントリ。ELF の `DT_RUNPATH` に相当。バイナリに複数存在できる |
| インストール名 | `LC_LOAD_DYLIB` に記録されているライブラリ識別子。例: `/usr/lib/libSystem.B.dylib`, `@rpath/libFoo.dylib` |
| `@executable_path` | インストール名に含まれるトークン。バイナリ本体（実行バイナリ）のディレクトリに展開される。ELF の `$ORIGIN` に相当 |
| `@loader_path` | インストール名に含まれるトークン。そのインストール名を持つ Load Command を含む Mach-O（バイナリまたは別の `.dylib`）のディレクトリに展開される |
| `@rpath` | インストール名に含まれるトークン。`LC_RPATH` エントリを順に展開して解決されるプレースホルダ |
| dyld shared cache | macOS 11 以降でシステム `.dylib`（`libSystem.B.dylib` 等）を単一のキャッシュファイルに統合した機構。ファイルシステム上に個別ファイルが存在しない場合がある |
| dyld shared cache ライブラリ | システムライブラリ系のインストール名を持ち、かつハッシュ計算可能な個別ファイル実体が利用できないライブラリ（§3.1.5 参照）。本タスクではハッシュ検証をスキップしコード署名検証に委譲する |
| `ImportedLibraries()` | `debug/macho` パッケージの `File.ImportedLibraries()`。`LC_LOAD_DYLIB` / `LC_LOAD_WEAK_DYLIB` のインストール名一覧を返す |
| `DynLibDeps` | `fileanalysis.Record` に保存される依存ライブラリスナップショット。ELF 版と共通フィールドを使用する |

## 3. 機能要件

### 3.1 ライブラリパス解決機能

#### FR-3.1.1: 依存ライブラリ名の取得

Mach-O バイナリの Load Commands から、依存ライブラリごとのインストール名と
Load Command 種別（`LC_LOAD_DYLIB` / `LC_LOAD_WEAK_DYLIB`）を取得できること。
実装は `debug/macho` パッケージの `macho.File.Loads` 等を走査し、各 Load Command から
インストール名とコマンド種別を対応付けて取得すること。

`macho.File.ImportedLibraries()` は依存ライブラリ名の一覧取得には利用できるが、
`LC_LOAD_DYLIB` と `LC_LOAD_WEAK_DYLIB` を区別できないため、これのみを依存種別の
判定に用いてはならない。別手段で weak/strong を同等に判定できる場合はその実装でもよい。

**補足**: FR-3.1.7 の「強い依存はエラー／弱い依存はスキップ」を実装するため、
依存ライブラリごとに weak/strong の属性を保持したまま後続の解決処理へ渡すこと。

#### FR-3.1.2: ライブラリパスの解決順序

インストール名から実際のファイルパスを決定する際、以下の順序で解決すること：

1. **絶対パス**（`@` トークンなし、`/` で始まる）: そのパスをそのまま使用する
2. **`@executable_path` トークン**: バイナリ本体のディレクトリに展開してパスを構築する
3. **`@loader_path` トークン**: そのインストール名を持つ Load Command を含む Mach-O
   のディレクトリに展開してパスを構築する
4. **`@rpath` トークン**: 当該 Mach-O の `LC_RPATH` エントリを順に試し、
   最初に実体ファイルが存在するパスに展開する
5. **デフォルト検索パス**: `/usr/local/lib`、`/usr/lib`（この順）で検索する
   （`DYLD_FALLBACK_LIBRARY_PATH` のデフォルト値 `/usr/local/lib:/usr/lib` に準拠。
   2023 年秋以降のバイナリでは dyld のデフォルトが存在しないが、本実装では
   バイナリの世代によらず常にこの順序で検索する。現代のツールチェーンでは
   インストール名に必ず絶対パスか `@` トークンが使われるため、step 5 に到達するケース
   自体が極めて稀であり、実用上の影響はほぼない）

**`DYLD_LIBRARY_PATH` の扱い**:

`record` 時は `DYLD_LIBRARY_PATH` / `DYLD_FALLBACK_LIBRARY_PATH` を**使用しない**。
これらはユーザーの実行環境に依存し、整合性検証の基準として不適切なためである。

`runner` は子プロセス起動前に `DYLD_LIBRARY_PATH` / `DYLD_FALLBACK_LIBRARY_PATH` を
**常にクリアする**（既存動作）。これにより環境変数によるパスハイジャック攻撃は
適用外となる。

#### FR-3.1.3: `@executable_path` / `@loader_path` / `@rpath` のトークン展開

インストール名の先頭に `@executable_path`、`@loader_path`、`@rpath` が含まれる場合、
それぞれ以下のように展開すること：

- **`@executable_path`**: 解析対象の**メインバイナリ**のディレクトリパスに置換する。
  例: `/opt/app/foo` が解析対象で、インストール名が `@executable_path/../lib/libfoo.dylib`
   の場合 → `/opt/app/../lib/libfoo.dylib` = `/opt/lib/libfoo.dylib` として解決する
- **`@loader_path`**: そのインストール名を持つ Load Command を含む **Mach-O ファイル**
   のディレクトリパスに置換する。メインバイナリの直接依存を解決する場合はメインバイナリの
   ディレクトリが基準となる。間接依存（FR-3.1.9）を解決する場合は、その Load Command を
   含む `.dylib` 自身のディレクトリが基準となる
- **`@rpath`**: `LC_RPATH` の各エントリを展開後、実際にファイルが存在するパスを採用する

**`LC_RPATH` における `@executable_path` / `@loader_path` の展開**:
`LC_RPATH` エントリ自体に `@executable_path` / `@loader_path` が含まれる場合も
同様に展開してから `@rpath` 解決に使用する。

#### FR-3.1.4: パスの正規化

ライブラリパスの記録および比較を行う際、すべてのパスを以下の手順で正規化すること：

1. `filepath.EvalSymlinks` でシンボリックリンクを解決し、物理的な実体パスを得る
2. `filepath.Clean` でパスを正規化する（`..`、重複スラッシュ等の除去）

この正規化は以下のタイミングで適用する：
- `record` 時: 各 `LibEntry.Path` を記録する前
- `runner` 実行時: ハッシュ計算前に記録済みの `LibEntry.Path` を正規化する

`filepath.EvalSymlinks` はファイルが実際に存在する場合のみ成功する。
`record` 時には解決済みライブラリが存在することが前提であり、問題ない。

#### FR-3.1.5: dyld shared cache ライブラリの判定とスキップ

ライブラリが、dyld shared cache に収録される可能性が高いシステムライブラリであり、
かつハッシュ計算可能な個別ファイル実体を取得できない場合、ハッシュ検証をスキップして
コード署名検証に委譲する。

**判定基準（dyld shared cache ライブラリ）**:
以下の 2 条件を満たす場合、dyld shared cache ライブラリとして扱う：

1. インストール名が以下のいずれかのシステムライブラリ系プレフィックスに一致する
   - `/usr/lib/` または `/usr/libexec/`
   - `/System/Library/`
   - `/Library/Apple/`
2. 解決後のパスに対してハッシュ計算可能な個別ファイル実体を取得できない

単にファイルシステム上に実体が存在しないだけの非システムライブラリは、dyld shared cache
ライブラリとはみなさず、
FR-3.1.7 の解決失敗として扱う。

**スキップ時の動作**:
- `DynLibDeps` にエントリを記録しない（ハッシュ検証の対象外）
- `slog.Info` レベルで「dyld shared cache 由来と判断し、コード署名検証に委譲した旨」と
   インストール名をログに出力する

**runner 時の動作**:
dyld shared cache ライブラリは `DynLibDeps` に記録されないため、
`runner` 側でも特別な処理は不要である。

#### FR-3.1.6: 未知トークンの扱い

`@executable_path` / `@loader_path` / `@rpath` 以外の `@` プレフィックスを持つトークン
（例: `@loader_rpath`, `@rpath_fallback` 等の将来拡張または非標準トークン）を検出した場合：

- `record` は継続する
- `AnalysisWarnings` に未知トークンとインストール名を含む警告を追加する
- 当該ライブラリはハッシュ検証不能として `DynLibDeps` に追加しない

これにより、親タスク 0095 の fail-safe 方針に合わせて、記録自体は継続しつつ
「ハッシュ検証不能な依存がある」ことを high risk 信号として保持できる。

#### FR-3.1.7: 解決失敗時の動作

| 依存ライブラリの種別 | 解決失敗時の動作 |
|---------------------|----------------|
| `LC_LOAD_DYLIB`（強い依存） | `record` コマンドをエラーで終了し、記録を行わない。エラーメッセージに解決できなかったインストール名を含める |
| `LC_LOAD_WEAK_DYLIB`（弱い依存） | そのライブラリの `DynLibDeps` へのエントリ追加をスキップし、`record` を継続する。`slog.Info` でスキップした旨をログに出力する |
| dyld shared cache ライブラリ（FR-3.1.5） | ハッシュ検証をスキップし、`record` を継続する |

**設計方針**: `LC_LOAD_DYLIB` の解決失敗で記録を失敗させることで、
不完全なスナップショットが保存されないようにする（ELF 版と同一方針）。

#### FR-3.1.8: Fat バイナリの扱い

Fat バイナリの場合、実行ホストのアーキテクチャに対応する `CPUType` と一致するスライスを
対象にライブラリ解決を行う。実装では `runtime.GOARCH` に対応する `CPUType` を求め、
`debug/macho.FatFile` から一致するスライスを選択し、その `LC_LOAD_DYLIB` /
`LC_RPATH` を使用する。

一致するスライスが存在しない場合は、別スライスへ暗黙にフォールバックせず、
`record` を失敗させる。これにより、実行ホストと異なるアーキテクチャの `LC_RPATH` /
`LC_LOAD_DYLIB` を誤って記録することを防ぐ。

**根拠**: Fat バイナリの依存ライブラリ一覧は全スライスで共通であることがほとんどだが、
`LC_RPATH` は各スライスで異なる場合がある。実行ホストのアーキテクチャに一致する
スライスのみを使用することで解決を単純化し、Intel macOS と Apple Silicon macOS の
いずれでも要件を明確にできる。

#### FR-3.1.9: 間接依存ライブラリの再帰的解決

直接依存として解決された各 `.dylib` に対して、さらにその `LC_LOAD_DYLIB` /
`LC_LOAD_WEAK_DYLIB` を読み取る再帰的な解決（BFS）を行い、完全な依存ツリーを
構築すること。

**再帰解決の手順**:
1. バイナリの `LC_LOAD_DYLIB` / `LC_LOAD_WEAK_DYLIB` から直接依存ライブラリを取得する
2. 各依存ライブラリを FR-3.1.2 のパス解決で `resolved_path` を求める
3. `LibEntry { SOName: install_name, Path: resolved_path, Hash: ... }` を記録する
4. 解決済みの `.dylib` をキューに追加し、BFS でさらにその依存を解決する
5. 解決済みパス（`visited` セット）を管理し、同一パスを 2 度以上解析しない

**`@rpath` 解決コンテキスト**:
間接依存ライブラリのパス解決（FR-3.1.2）は、各 `.dylib` 自身の `LC_RPATH` を用いて行う。
`@executable_path` および `@loader_path` は「そのインストール名を持つ Load Command を含む
Mach-O ファイル」のディレクトリを基準として展開する（FR-3.1.3 の定義と同一）。
間接依存の解決時には、メインバイナリではなく当該 `.dylib` 自身のディレクトリが基準となる。

**dyld shared cache ライブラリのスキップ**:
FR-3.1.5 の判定を間接依存にも適用する。スキップされたライブラリはキューに追加しない
（その先の依存は解決不要）。

**解決深度の上限**:
再帰深度の上限を 20 段とすること。上限超過時は `record` をエラーで終了し、記録を行わない。
通常の macOS バイナリの依存深度は数段であるため、上限超過は異常な構成を示す。

**セキュリティ上の意義**:
直接依存のみの検証では、サードパーティライブラリやバンドル `.dylib` の間接依存が
差し替えられた場合に検出できない。BFS による完全な依存ツリーの検証により、このような
深い位置での差し替えも検出する。

### 3.2 依存ライブラリのハッシュ記録（`record` 拡張）

#### FR-3.2.1: 既存 `DynLibDeps` フィールドの使用

`fileanalysis.Record.DynLibDeps`（`[]fileanalysis.LibEntry`）は ELF 版タスク 0074 で
既に定義されている。Mach-O 版でも同一フィールドを使用する。
**別フィールドは追加しない**（タスク 0095 §9 スキーマ方針に準拠）。

各エントリのフィールドの意味は以下の通り：

| フィールド | Mach-O での意味 |
|-----------|----------------|
| `SOName`  | `LC_LOAD_DYLIB` に記録されているインストール名（例: `@rpath/libFoo.dylib`） |
| `Path`    | 解決・正規化されたフルパス（例: `/opt/app/lib/libFoo.dylib`） |
| `Hash`    | `"sha256:<hex>"` 形式のハッシュ値 |

#### FR-3.2.2: `record` 時の拡張

`record` コマンドがコマンドのハッシュを計算・保存する際に、
Mach-O バイナリであれば `DynLibDeps` も合わせて記録すること。

- バイナリが非 Mach-O、または `LC_LOAD_DYLIB` / `LC_LOAD_WEAK_DYLIB` エントリがない
  場合は `DynLibDeps` を記録しない（`omitempty` により JSON に出力されない）
- すべての依存ライブラリが dyld shared cache ライブラリのみの場合も `DynLibDeps` は nil
- 既存の `ContentHash`、`SymbolAnalysis`、`SyscallAnalysis` フィールドは変更しない
- 未知 `@` トークン等でハッシュ検証不能な依存があった場合は、`DynLibDeps` とは別に
   `AnalysisWarnings` へ記録する

#### FR-3.2.3: `--force` フラグとの整合性

`record --force` 実行時は `DynLibDeps` も新しい値で上書きする。

#### FR-3.2.4: スキーマバージョン

本タスクで**スキーマバージョンを 13 から 14 に更新する**。

バージョン 14 以降の記録では、Mach-O バイナリについても `DynLibDeps` の有無を
ELF 版と同等に検証する（FR-3.3.2 参照）。バージョン 13 以前の記録（本タスク導入前）は
後方互換のため旧ルールで扱う（NFR-4.3 参照）。

既存の ELF 版 `DynLibDeps` 記録はバージョンアップ後も変更なく使用できる。

### 3.3 依存ライブラリの整合性検証（`runner` 拡張）

#### FR-3.3.1: `runner` 実行時の検証方式

`runner` がコマンドを実行する前に `filevalidator.Validator.Verify()` でハッシュ検証を行う
既存フローの中で、Mach-O バイナリかつ `DynLibDeps` が記録されているコマンドに対しても
ライブラリの整合性を検証する。

検証は**ハッシュ照合のみ**（1 段階）で行う：

`DynLibDeps` に記録された各 `LibEntry.Path` のファイルをハッシュ計算し、
`LibEntry.Hash` と比較する。これにより `record` 時に使用されていたライブラリ自体が
改ざん・差し替えされていないことを確認する。

`runner` は `DYLD_LIBRARY_PATH` / `DYLD_FALLBACK_LIBRARY_PATH` を実行前に常にクリアするため、
パスハイジャック攻撃は適用外となる。

以下の攻撃ベクトルに対する防御：

| 攻撃ベクトル | 対応 |
|------------|------|
| ライブラリ自体の改ざん（`record` 時と同じパスで中身を書き換え） | ハッシュ照合で検出 |
| ライブラリ丸ごと差し替え（同一パスへ別ライブラリをコピー） | ハッシュ照合で検出 |
| パスハイジャック（`DYLD_LIBRARY_PATH` 等で別ライブラリにリダイレクト） | runner 側で環境変数クリア済みのため適用外 |

#### FR-3.3.2: 検証の判定ロジック

| 状態 | 判定 | 動作 |
|------|------|------|
| `DynLibDeps` が記録されておらず、対象が非 Mach-O バイナリ | 正常 | 従来通り実行を許可 |
| `DynLibDeps` が記録されておらず、対象が Mach-O バイナリ、かつスキーマバージョン < 14 | 正常 | 実行を許可（本タスク導入前の旧記録との後方互換性。§4.3 参照） |
| `DynLibDeps` が記録されておらず、対象が Mach-O バイナリ、かつスキーマバージョン ≥ 14 | エラー | 実行をブロック。`record` の再実行を要求するエラーを返す |
| 全ライブラリについてハッシュが一致 | 正常 | 実行を許可 |
| ハッシュ不一致 | エラー | 実行をブロック。`record` の再実行を要求するエラーを返す |
| `path: ""` のエントリが存在する | エラー | 実行をブロック。不完全な記録として `record` の再実行を要求するエラーを返す |

**ELF 版との統一**: スキーマバージョン 14 以降の記録では、ELF 版（タスク 0074 FR-3.3.2）と
同様に Mach-O バイナリで `DynLibDeps` が欠落している場合もエラーとする。
旧スキーマ（< 14）の記録のみ後方互換のため正常扱いとする。

**補足**: `DynLibDeps` が空でも、`record` 時に未知トークン等の理由でハッシュ検証不能な依存が
`AnalysisWarnings` に記録されている場合は、本 dynlib 検証とは別に既存の high risk 判定系で
扱う。つまり「`DynLibDeps` が空であること」自体を Mach-O で常に安全とみなすわけではない。

#### FR-3.3.3: エラーメッセージ

ライブラリ不一致検出時のエラーメッセージは以下の情報を含むこと：
- 不一致が検出されたライブラリのインストール名（`SOName`）
- ハッシュ不一致の場合: 期待されるハッシュ（記録時）と実際のハッシュ（現在）
- `record` の再実行を促すメッセージ

## 4. 非機能要件

### NFR-4.1: パフォーマンス

#### NFR-4.1.1: `record` 時の解析時間

ライブラリパス解決およびハッシュ計算は、一般的なバイナリ（直接依存ライブラリ数 5〜15 程度）
に対して実用的な時間内に完了すること。

#### NFR-4.1.2: `runner` 実行時のオーバーヘッド

ライブラリハッシュ検証は依存ライブラリ数に比例した時間がかかる。
dyld shared cache ライブラリはスキップするため対象ライブラリ数は限定的であり、
通常は数十 ms 以内に収まることを想定する。

タスク 0095 NFR-5.1 に従い、既存比で実行時間が **2 倍を超えない**こと。

### NFR-4.2: セキュリティ

#### NFR-4.2.1: 安全なファイル読み取り

ライブラリのハッシュ計算時は `safefileio` パッケージを使用し、
シンボリックリンク攻撃を防止すること。

#### NFR-4.2.2: パス解決の安全性

`@executable_path` / `@loader_path` / `@rpath` 展開やパス結合時には
ディレクトリトラバーサル対策（`filepath.Clean` による正規化）を行うこと。
ライブラリパスの記録・比較時には `filepath.EvalSymlinks` + `filepath.Clean` で正規化し、
シンボリックリンク由来の偽不一致を防ぐこと（FR-3.1.4 参照）。

### NFR-4.3: 後方互換性

本タスクで**スキーマバージョンを 13 から 14 に更新する**（FR-3.2.4 参照）。

スキーマバージョン 13 以前の既存記録（本タスク導入前に `record` した Mach-O バイナリ）は
`DynLibDeps` を持たないが、`runner` は正常扱いとし実行をブロックしない（FR-3.3.2 参照）。

スキーマバージョン 14 以降の新規記録では `DynLibDeps` が必須となるため、
本タスク導入後に `record` を実行し直すことでハッシュ検証が有効になる。

### NFR-4.4: 保守性

#### NFR-4.4.1: 外部依存なし

`otool` コマンド等の外部ツールに依存しない。
Go 標準ライブラリ（`debug/macho`）のみを使用する。

#### NFR-4.4.2: machoanalyzer との共存

本タスクの実装では、Load Command 走査や `@rpath` 展開のような Mach-O 固有で再利用可能な
処理を既存の Mach-O 系コードと重複実装しない。再利用可能部分は `machoanalyzer` または
隣接する共有ユーティリティに集約し、dynlib 整合性検証固有の処理のみを分離する。

### NFR-4.5: テスタビリティ

Mach-O バイナリの Load Command 解析（`LC_LOAD_DYLIB`、`LC_RPATH` 等）は
macOS 環境でのみ動作するため、テストは macOS 上で実行できる必要がある。

テスト用 Mach-O フィクスチャ（`LC_RPATH`、`@rpath` インストール名を持つ最小バイナリ）を
リポジトリに含める。フィクスチャ生成スクリプトを `testdata/` に格納し再現性を確保する。

## 5. 受け入れ条件

### AC-1: ライブラリパス解決

- [ ] `LC_LOAD_DYLIB` エントリのないバイナリ（静的リンク等）では `DynLibDeps` が記録されないこと
- [ ] 絶対パスのインストール名（`/usr/local/lib/libFoo.dylib` 等）が正しく解決されること
- [ ] `@executable_path` トークンがメインバイナリのディレクトリに展開されること
- [ ] `@loader_path` トークンがそのロードコマンドを持つバイナリのディレクトリに展開されること

- [ ] `LC_RPATH` に `@executable_path` を含むエントリが正しく展開されること
- [ ] デフォルト検索パス（`/usr/local/lib`, `/usr/lib` の順）からライブラリが解決されること

- [ ] ファイルシステム上に存在しないライブラリがスキップ（`LC_LOAD_WEAK_DYLIB`）またはエラー（`LC_LOAD_DYLIB`）になること
- [ ] 未知の `@` トークンを含むインストール名で `record` は継続し、`AnalysisWarnings` に警告が記録され、当該ライブラリが `DynLibDeps` に含まれないこと
- [ ] `LC_LOAD_DYLIB` の解決失敗で `record` がエラーで終了し、記録が保存されないこと

- [ ] 循環依存が生じても無限ループせず正常終了すること
- [ ] 再帰深度が上限（20 段）を超えた場合に `record` がエラーで終了し、記録が保存されないこと

## 6. テスト方針

### 6.1 ライブラリパス解決のユニットテスト

| テストケース | 検証内容 |
|-------------|---------|
| 絶対パスのインストール名 | そのパスが直接使用されること |
| `@executable_path` 展開 | メインバイナリのディレクトリへの展開 |
| `@loader_path` 展開 | ローダーのディレクトリへの展開 |
| `@rpath` 展開（単一 `LC_RPATH`） | `LC_RPATH` エントリへの展開 |
| `@rpath` 展開（複数 `LC_RPATH`） | 最初に存在するパスへの展開 |
| `LC_RPATH` に `@executable_path` | 正しく二重展開されること |
| dyld shared cache ライブラリ | スキップされること |
| ファイルシステム上に存在しない `@rpath` | 次の `LC_RPATH` エントリへのフォールバック |
| `LC_LOAD_DYLIB` 解決失敗 | `record` がエラーで終了し記録が保存されないこと |
| `LC_LOAD_WEAK_DYLIB` 解決失敗 | スキップして `record` が継続されること |
| 未知 `@` トークン | `record` は継続し、`AnalysisWarnings` に記録されること |
| デフォルトパス検索 | `/usr/local/lib`, `/usr/lib` の順で解決 |
| 間接依存の再帰解決 | 直接依存の `.dylib` がさらに依存する `.dylib` も記録されること |
| 間接依存での `@rpath` 解決 | 各 `.dylib` 自身の `LC_RPATH` が使われること（メインバイナリの `LC_RPATH` ではない） |
| 循環依存の防止 | シンボリックリンク等で循環が生じても無限ループしないこと |
| 再帰深度上限 | 深度上限超過時に `record` がエラーで終了し記録が保存されないこと |

### 6.2 `record` / `runner` の統合テスト

| テストケース | 検証内容 |
|-------------|---------|
| 正常ケース | `record` 後に `runner` で検証成功（ハッシュ一致） |
| ライブラリ改ざん | `record` 後にライブラリの中身を書き換えてハッシュ不一致を検出 |
| ライブラリ差し替え | `record` 後に同一パスへ別ライブラリをコピーしてハッシュ不一致を検出 |
| 非 Mach-O バイナリ（スクリプト等） | `DynLibDeps` なしで正常動作 |
| 既存 Mach-O 記録（`DynLibDeps` なし） | 後方互換性: 実行が許可されること |
| `LC_LOAD_DYLIB` 解決失敗（`record` 時） | `record` がエラーで終了し記録が保存されないこと |
| 未知 `@` トークン | `record` は継続し、warning が保存されること |

### 6.3 テストフィクスチャ

`@rpath` インストール名と `LC_RPATH` エントリを持つ最小の Mach-O バイナリを
フィクスチャとしてリポジトリに含める（`testdata/` ディレクトリ）。
フィクスチャ生成スクリプトを合わせて格納し再現性を確保する。

## 7. 設計上の制約と限界

1. **間接依存の BFS 再帰解決**（FR-3.1.9）: ELF 版タスク 0074 と同様に、BFS による完全な
   依存ツリーを検証対象とする。macOS では間接依存の大半は dyld shared cache に収録される
   システムライブラリ（FR-3.1.5 でスキップ）であるため、実際に BFS が展開するのは
   Homebrew やバンドル `.dylib` などサードパーティライブラリのサブツリーに限定される。

2. **dyld shared cache ライブラリのハッシュ検証不能**: `libSystem.B.dylib` 等のシステムライブラリは
   macOS 11 以降ではファイルシステム上に実体が存在しない場合がある。これらは
   Apple のコード署名検証に委譲するため、本システムではサプライチェーン攻撃として扱わない。
   dyld shared cache からの抽出によるハッシュ検証はタスク 0100 の扱い。

3. **`DYLD_INSERT_LIBRARIES` 注入攻撃の非検出**: `DYLD_INSERT_LIBRARIES` による
   任意 `.dylib` の注入は `DynLibDeps` には記録されない。`runner` は実行前にこの
   環境変数をクリアするため（既存動作）、攻撃は実質的に無効化されている。

4. **`dlopen()` の引数による実行時ロード**: `dlopen` に渡されるライブラリ名は
   実行時に決定されるため静的解析では特定できない。タスク 0074 方策 B
   （`dlopen` シンボル検出）により既に対応済み。

5. **Fat バイナリのスライス選択**: 本タスクではネイティブスライスのみを対象とする。
   非ネイティブスライスの `LC_LOAD_DYLIB` が異なる場合は検証対象外となる。
   クロスアーキテクチャ攻撃対策は `StandardMachOAnalyzer` のシンボル解析層で
   全スライスを解析する既存実装に委ねる。

6. **間接依存の `@rpath` 解決における `LC_RPATH` 非継承**: 間接依存ライブラリのパス解決では、
   各 `.dylib` 自身の `LC_RPATH` エントリのみを使用し、メインバイナリ（または上位ローダー）の
   `LC_RPATH` は引き継がない。

   **実際の macOS dyld との相違**: macOS の dyld は `@rpath` 解決時にロードチェーン全体の
   `LC_RPATH` を蓄積して使用する（メインバイナリ → 各中間 `.dylib` の順で蓄積）。これは ELF の
   `DT_RPATH` に近い継承動作であり、非継承の `DT_RUNPATH` とは異なる。本実装は dyld の実際の
   動作とは一致しない。

   **影響を受けるケース**: メインバイナリの `LC_RPATH`（例: `@executable_path/../Frameworks`）に
   依存して `@rpath` を解決する間接依存ライブラリが存在する場合、`record` 時に解決失敗エラーと
   なる。これは App Bundle 構造（`*.app/Contents/Frameworks/` 内のフレームワークが
   バンドル共有 `.dylib` を `@rpath` で参照するパターン）で発生しうる。

   **現実的な影響の限定**: macOS の一般的なプラクティスでは、自己完結型フレームワーク
   （`*.framework`）は自身の `LC_RPATH` に `@loader_path/` や `@loader_path/../` を含め、
   ロードチェーン継承に依存しない設計を推奨している。また、本実装での解決失敗は
   `LC_LOAD_DYLIB`（強い依存）の場合は `record` をエラーで終了させるため、
   誤った（不完全な）スナップショットが保存されることはなく、問題は明示的に検出される。

   **将来的な対応方針**: ロードチェーン継承による `@rpath` 解決（BFS キューに蓄積 rpaths を
   持たせる実装）は、実際に継承が必要なケースが確認された時点でタスク化して対応する（YAGNI）。
